# 通用规范与模板

## 任务类型
- spec 任务：只负责实现设计文档，将功能以及边界，测试标准说清楚。
- builder 任务：根据spec 实现的设计文档，进行代码实现与测试。
- review 任务：builder 任务完成后，review 设计文档和代码实现的缺陷，然后给出修改建议 review任务需要仔细纠错，并可补充认为必要的功能建议。
- 合并任务，审查通过后，如果任务在某个 worktree上，那么合并到主分支并删除工作树。
- 其他任务：其他简单任务。
- 注意事项：
  - spec 任务只写 spec 文件，不做实现。
  - 实现任务只负责实现，测试。
  - 审查任务只审查给意见，不写代码。
  - 不要一个任务同时做 spec、实现、审查或合并。

## 任务工作流
- 默认流程：spec 任务 -> 实现任务（含测试） -> 审查任务。
- 审查不通过时，按问题回到实现任务（含测试）再次迭代，直到审查通过，审查不可一次通过，需要反复修改才能得到好的作品。
- 审查通过后再创建合并任务，将工作树改动合入主分支。
- 审查任务需要仔细纠错，并可补充认为必要的功能建议。

## 审查标准
- 必须覆盖需求与 spec 的一致性。
- 必须指出潜在风险或边界条件缺失。
- 必须检查脚本参数与路径使用是否符合规范。
- 必须说明是否需要新增或更新测试。

## 合并标准
- 审查通过且无未解决阻塞问题。
- 合并任务明确工作树与目标分支。
- 合并后删除对应工作树。
- 合并结果需回报管理员并记录日志路径。

## 合并任务细则
- 合并任务只做合并与清理，不补功能、不改需求、不写测试。
- 合并前再次确认工作树是目标任务对应的独立 worktree。
- 合并后记录：合并时间、分支、工作树路径、涉及任务 ID、日志路径。
- 合并完成后删除对应 worktree，避免残留。

## 回报格式模板
- 任务完成回报应包含：任务 ID、完成时间、变更说明、日志路径、后续计划。
- 示例（按实际替换占位符）：

```
任务 ID: <task-id>
完成时间: <yyyy-mm-dd hh:mm>
变更说明: <summary>
日志路径: <log-path>
后续计划: <next-step or none>
```

## 审查意见模板
- 审查意见需包含：结论、问题清单、风险说明、建议修复项。
- 示例（按实际替换占位符）：

```
结论: 通过 / 不通过
问题清单:
- <issue-1>
- <issue-2>
风险说明: <risk-summary>
建议修复项:
- <fix-1>
- <fix-2>
```

## 任务分配模板
- 任务分配信息需包含：任务 ID、任务类型、目标角色、工作树要求、交付物。
- 示例（按实际替换占位符）：

```
任务 ID: <task-id>
任务类型: spec / 实现 / 审查 / 合并
目标角色: <role>
工作树要求: <worktree-path or create-new-worktree>
交付物: <spec file / code change / review report / merge log>
```

## 合并回报模板
- 合并回报需包含：任务 ID、合并时间、分支、工作树路径、变更范围、日志路径。
- 示例（按实际替换占位符）：

```
任务 ID: <task-id>
合并时间: <yyyy-mm-dd hh:mm>
目标分支: <branch>
工作树路径: <worktree-path>
变更范围: <summary>
日志路径: <log-path>
```

## 合并前检查清单
- 任务已通过审查且无阻塞问题。
- 工作树与任务 ID 匹配且为独立 worktree。
- 变更范围与任务一致，无额外未说明改动。
- 必要测试已执行或说明不执行的原因。
- 合并目标分支正确且已确认。

## 任务验收模板
- 任务验收需包含：任务 ID、验收结论、验收依据、遗留问题（如有）、后续动作。
- 示例（按实际替换占位符）：

```
任务 ID: <task-id>
验收结论: 通过 / 不通过
验收依据: <spec / test / review>
遗留问题: <none or list>
后续动作: <next-step or none>
```
