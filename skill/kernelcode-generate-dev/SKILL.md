# kernelcode_generate 开发指导

## 使用场景
- 用户请求与 `kernelcode_generate` 的功能开发、重构、测试、调试、文档维护相关。
- 任务目标是“可运行结果”，而不是只做概念性讨论。

## 开始前先读
- `references/dev-checklist.md`：开发检查清单（需求确认、实现门禁、测试门禁、交付门禁）。
- 项目根目录 `ARCHITECTURE.md`：当前目录结构和建议分层。

如果用户任务只涉及单文件小修（例如文案/注释修正），可以跳过深度架构审阅；其他任务默认先读上述两份文件。

## 工作流程（默认）
1. 识别目标与边界
- 明确用户可见输出（例如生成 C/CUDA 代码、补测试、修 DSL 行为）。
- 明确是否要求兼容旧行为；若未说明，默认优先兼容。
- 列出影响层：`dsl`、`ir`、`compiler/pass_manager`、`target`、`tests`、`docs`。

2. 探测仓库现状
- 先用快速命令查看真实目录与文件，不假设建议结构已经全部落地。
- 若项目仍是骨架状态，优先创建最小可运行闭环（最少必要文件 + 示例 + 测试）。

3. 实施改动
- 按“前端 DSL -> IR -> Pass -> 后端发射器”的方向推进，保持语义单向传递清晰。
- 新增能力时，同步补示例和测试；避免只改一层导致系统不闭合。
- 后端发射器输出必须可复现（同输入得到稳定、确定的文本输出）。

4. 验证
- 优先运行定向测试；有条件时再跑全量测试。
- 至少验证一条主路径；若涉及异常分支，增加边界/错误路径验证。
- 无法运行测试时，明确说明原因、未验证范围与风险。

5. 交付
- 汇总改动文件、行为变化、测试命令与结果。
- 若存在兼容性风险或后续工作，明确列出 next steps。

## 实现准则
- 向后兼容优先：除非用户明确允许破坏式变更。
- IR 命名一致：节点/字段命名保持可读且语义稳定。
- 分层职责清晰：避免把后端细节泄漏到 DSL 层。
- 最小改动原则：只改与目标直接相关的文件，避免无关重构。
- 文档同步：外部行为变化时更新 `README.md` 或架构说明。

## 测试策略
- 小改动：运行受影响模块的最小测试集。
- 行为变化：补充回归测试，覆盖新行为与旧行为兼容性。
- 后端相关：断言关键生成代码片段（函数签名、内核主体、launch/config 关键字段等）。

## 输出要求（回复用户时）
- 先给结果结论，再给关键实现细节。
- 必须包含：修改文件列表、执行过的命令、测试结果。
- 若有未完成项，明确标注阻塞原因与建议下一步。
