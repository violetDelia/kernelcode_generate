# 异常处理规范

## 文件说明
- 功能说明：统一阻塞、待确认、范围冲突和权限问题的处理方式。
- 使用示例：遇到不能继续推进的情况，先按本文件分类，再写记录和回报。

## 通用原则
- 普通质量问题不暂停，直接写成返工项；只有执行链无权或无法处理的问题才暂停。
- 阻塞先写记录，再回报管理员、架构师或用户。
- 流程、权限、文件范围、任务脚本问题问管理员。
- 实现边界、接口语义、验收口径、架构争议问架构师。
- 公开 `API` 变化、需求取舍和 `expectation` 修改授权问用户。
- 必过合同失败先复现并记录 actual / expected / spec / verdict；能由实现、`spec`、测试或任务记录收口时直接返工，只有涉及 `expectation/` 本体、未确认公开 `API` 或角色权限无法裁定时才暂停。

## 常见异常
| 异常 | 处理 |
| --- | --- |
| 任务目标泛化 | 要求改成可执行动作、对象和验收结果 |
| `worktree` 或记录路径不清 | 暂停并问管理员 |
| 公开 `API` 变更缺用户确认 | 暂停并交架构师整理待确认项 |
| 需要跨文件调用非公开 API | 暂停并交架构师裁定公开边界 |
| 测试需要直连非 API helper | 改测公开 API；不能改时请求架构裁定 |
| `expectation` 本体需要调整 | 执行链暂停并记录证据，只允许用户或架构师按授权范围处理 |
| 可读性、可维护性或测试有效性不足 | 不暂停，写成返工项并回到对应角色处理 |
| 验收失败但疑似环境问题 | 记录命令、退出码、环境和最小复现 |
| 发现无关本地改动 | 不回退；若冲突则询问用户或管理员 |

## 阻塞记录
- 写清阻塞类型、复现命令、失败摘要、已排除项、建议接手角色。
- 待用户确认项必须写清选项、影响和推荐项。
- 不得用口头回报替代记录。
