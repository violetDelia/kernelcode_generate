# 测试文件约定

## 文件说明
- 功能说明：统一测试文件组织、公开 API 测试边界、diff 反推测试和 `expectation` 关系。
- 使用示例：新增或修改测试前，先按本文件确认测试是否越界。

## 基本规则
- 测试必须验证公开 `API`、公开行为、稳定错误语义或明确的命令入口。
- 测试不得跨文件直连非公开 helper。
- 若公开行为无法通过现有 API 触达，先回到 `spec` 和架构裁定，不用私有入口绕过。
- 测试文件命名和目录应跟被测模块对应。
- 断言必须能在实现错误时失败；不得只断言“无异常”。
- 测试应像可执行文档一样说明行为：输入、动作、预期结果和失败语义必须清楚。
- 优先覆盖边界、错误路径、回归点和最小公开合同；不得只验证 happy path。
- 测试结构保持直接、可读、确定；过度 mock、过度参数化或复制实现逻辑的测试应返工。

## API 边界
- 测试可导入 `spec` 和文件级 `API 列表` 声明的公开入口。
- 当前测试文件内部可以定义本地 helper。
- 跨文件共享测试 helper 必须是测试目录公开约定的一部分，且不得包装业务非公开 API。
- 不能为了覆盖率把非公开实现细节升级成测试入口。

## diff 反推测试
- 修改实现时，必须按实际 diff 反推测试。
- 修改 `spec` 时，必须跑能锁定对应公开合同的测试或文本核对脚本。
- 修改工具入口时，必须覆盖 CLI、参数、输出和失败语义。
- 修改 include 或运行时源码时，必须覆盖编译、运行或静态检查入口。
- 重构类改动必须证明公开行为保持不变；若只声称“整理结构”但测试不能锁定行为，不得通过。
- 修改计划、标准或提示词时，至少执行文本残留扫描和格式检查。

## expectation 关系
- `expectation` 是合同验收资产，不是 diff 反推测试。
- 终验和复验运行当前计划正文列为必过合同验收资产的相关 `expectation`；历史 / 本地只读来源说明不作为当前必跑测试。
- 执行和审查仍要补 pytest 或测试脚本。
- 测试不得为了匹配 `expectation` 输出而绕过公开 API。
- 若 `expectation` 本体疑似错误，非架构角色只能记录并请求架构裁定。

## 随机化测试
- 随机输入必须有稳定断言。
- 随机来源、范围和排除条件要可读。
- 重复运行失败时记录 seed、失败输入或可复现摘要。

## 记录要求
- 记录测试命令、退出码、通过数量或失败摘要。
- 未运行的相关测试必须写原因和风险。
- 随机化测试应写清随机来源、稳定断言和重复运行策略。
