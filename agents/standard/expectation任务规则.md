# expectation 任务规则

## 文件说明
- 功能说明：统一约束 `expectation` 文件的所有权、允许操作与合并边界，减少各角色提示词中的重复描述。
- 使用示例：各角色提示词可直接引用本文件，例如：`通用规则以 [expectation任务规则](../../../standard/expectation任务规则.md) 为准。`
- 使用示例：当执行人发现任务似乎需要改 `expectation` 时，先按本文件判断“是否必须转交架构师”，再决定是否继续。

## 通用规则
- `expectation` 文件是合同资产，默认由架构师维护；非架构师不得直接修改仓库中的 `expectation` 文件。
- 架构师修改 `expectation` 时，必须同步在计划书、专题 `spec` 或任务日志中写清修改原因、目标输出与对应验收资产。
- 非架构师若需参考 `expectation`，只能复制到临时位置做本地对照；不得修改原文件，不得把 `expectation` 改动纳入正常合并内容。
- 执行人、审查人、合并人在任务过程中若判断“当前需求必须改 `expectation` 才能继续”，应立即暂停当前阶段并询问架构师，不得自行扩范围。
- `李白` 合并时若发现变更中包含非架构师提交的 `expectation` 改动，应拒绝合并并回报管理员与架构师。
- 当任务本身就是 `expectation` 合同收口时，链路默认仍按 `spec -> build -> review -> merge` 推进，但真正修改 `expectation` 的阶段只能由架构师或被架构师明确授权的规格任务完成。
