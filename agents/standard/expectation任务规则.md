# expectation 任务规则

## 文件说明
- 功能说明：统一 `expectation/` 合同资产的所有权、允许操作、验收边界和合并规则。
- 使用示例：执行、审查、合并或终验时，先用本文件判断是否可以接触 `expectation/` 本体。

## 所有权
- `expectation/` 是合同资产，只允许用户或架构师在明确授权范围内修改、移动、重命名、新建或删除。
- 架构师修改 `expectation/` 前，必须写清用户确认来源、合同目标、影响范围、运行入口和预期输出。
- 执行、审查、合并、管理员和替补角色对 `expectation/` 只有读取、执行、引用和记录权限。
- 非架构角色不得把 `expectation` 改动纳入正常实现、审查或合并范围。

## 使用边界
- `expectation` 可作为合同真源和终验资产，但只有当前计划正文把它列为必过合同验收资产时，才是当前阻断项。
- 若计划正文把某个 `expectation` 收为历史 / 本地只读合同来源说明，或 latest main 明确不存在该资产且正文不再列为必过入口，只能记录为背景，不得当作当前阻断项。
- 当前必过 `expectation` 与实现冲突时，优先收口实现、`spec` 和测试。
- `expectation` 不计入 diff 反推测试；另跑时单列为“合同验收”。
- 非架构角色发现 `expectation` 本体缺口时，必须暂停并请求架构裁定，不得自行改文件。
- 架构裁定认为无需改 `expectation` 时，执行链只能按现有合同继续收口实现和测试。
- 只有用户明确要求全量 `expectation` 时，才把全量作为验收范围；否则只跑本轮相关入口。

## 计划与终验
- 计划书使用 `expectation` 作为合同时，必须写清合同真源顺序、入口命令、预期输出、是否只读，以及它是当前必过验收资产还是历史 / 本地只读来源说明。
- 终验、复验或修复复核必须在最新同步现场运行当前计划正文列为必过合同验收资产的相关 `expectation`。
- `expectation` 失败不得被口头忽略；若失败与本轮无关，必须写明基线证据和用户或架构认可口径。
- 历史说明、归档说明或对齐记录不等于当前合同资产；是否阻断只看当前计划正文、公开行为和当前验收资产。

## 合并边界
- 合并内容包含 `expectation/` 时，必须确认改动来自用户或架构师的明确授权。
- 缺少授权来源、合同目标或验收记录时，合并角色必须拒绝带入。
- 未被当前任务明确点名的 `expectation` 改动不得顺手合并。
