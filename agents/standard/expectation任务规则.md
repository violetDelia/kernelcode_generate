# expectation 任务规则

## 文件说明
- 功能说明：统一约束 `expectation` 文件的所有权、允许操作与合并边界，减少各角色提示词中的重复描述。
- 使用示例：各角色提示词可直接引用本文件，例如：`通用规则以 [expectation任务规则](../../../standard/expectation任务规则.md) 为准。`
- 使用示例：当执行人发现任务似乎需要改 `expectation` 时，先按本文件判断“是否必须转交架构师”，再决定是否继续。

## 通用规则
- `expectation` 文件是合同资产，默认对所有角色只读；未经用户明确授权，任何角色都不得直接修改、移动、重命名、删除或新建仓库中的 `expectation` 文件。
- 架构师也不得把“直接改 `expectation`”当作默认解法；若判断必须改 `expectation` 才能继续，必须先在计划书、专题 `spec` 或任务记录中写清阻塞原因，再询问用户是否放开。
- 各执行角色若需参考 `expectation`，只能读取、执行、引用或复制到临时位置做本地对照；不得修改原文件，不得把 `expectation` 改动纳入正常合并内容。
- 若 `expectation` 路径正确、入口可执行、输出明确，则默认它是当前合同真源；实现与其冲突时，执行人应优先修改实现与测试对齐，不因“实现不一致”自动升级为架构问题。
- 执行人、审查人、合并人在任务过程中若判断“当前需求必须改 `expectation` 本身、或当前 `expectation` 路径/入口/合同存在问题”才能继续，必须暂停当前阶段并询问管理员与架构师；未经用户明确授权，不得自行继续改 `expectation`。
- `李白` 合并时若发现变更中包含任何角色提交的 `expectation` 改动，应拒绝合并并回报管理员与架构师。
- 当任务本身看起来是 `expectation` 合同收口时，链路仍按 `spec -> build -> review -> merge` 推进，但默认只允许收口实现、`spec`、测试、计划资产与记录；`expectation` 文件本体仍保持只读，除非用户明确授权放开。
