# 任务新建模板

## 文件说明
- 功能说明：统一 `codex-multi-agents-task.sh -new -info` 的任务描述格式，减少不同角色之间的沟通歧义。
- 使用示例：`demo-plan-execute；计划书：《ARCHITECTURE/plan/demo.md》；任务目标：按计划书完成 demo 公开 API、实现、测试与验收资产闭环；任务链记录：20260405-demo.md`


## 通用规则
- 任务描述默认使用单行短格式，便于直接放入 `-new -info "<...>"`。
- 有计划书时，一个计划书只创建一个计划级大任务，默认写成 `"<plan>-plan-execute；计划书：《<计划书路径>》；任务目标：<...>；任务链记录：<...>"`。
- 计划书内部 `S1/S2/...` 是计划内小任务卡，不再放进 `-new -info` 作为独立任务编号。
- 计划内小任务卡必须在计划书正文里写清执行步骤；管理员分发时只引用整份计划，不把步骤压缩进 `-new -info`。
- 无计划书时，不写 `计划书` 字段，默认写成 `"<阶段>；任务目标：<...>；任务链记录：<...>"`。
- `阶段` 只使用：`execute`、`review`、`refactor`、`merge`、`other`；计划书默认使用 `execute`。
- `任务目标` 只写当前阶段直接产出，不写泛化目标或长期规划。
- `任务目标` 必须是执行人可直接照做的动作语句，推荐使用“动词 + 对象 + 验收结果”写法，例如：
  - `添加 xxx 接口，并跑通 pytest -q test/...`
  - `为 xxx 函数增加 yyy 功能，并跑通 python3 -m expectation...`
  - `修复 xxx 报错，使 xxx expectation 入口通过`
- `任务目标` 不得写成泛描述，例如：
  - `继续推进`
  - `按当前口径收口`
  - `整理一下结构`
  - `看下还有什么问题`
- 若任务由计划书派生，`任务目标` 应概括整份计划级大任务的直接产出，并指向计划书内的小任务卡；不得把 `S1/S2/...` 分别新建成多个独立任务。
- `任务链记录` 一律写当前链路沿用的记录文件短名。
- `<plan>` 只写计划简称或计划文件核心名，不重复整条路径；完整路径放在 `计划书：《...》` 中。
- 默认不把 `worktree`、`验收命令`、`验收标准` 塞进任务描述；只有管理员明确要求时才追加。
- 若计划书终验不通过，需要接续修复时，仍沿用原计划书路径和原计划级任务；`任务目标` 第一短句直接写 `修复：<最小阻断项>`，避免和正常阶段混淆。

## 推荐模板
有计划书：`<plan>-plan-execute；计划书：《<计划书路径>》；任务目标：<整份计划的规格、实现、测试与验收闭环>；任务链记录：<记录文件短名>`

无计划书：`<阶段>；任务目标：<当前阶段直接产出>；任务链记录：<记录文件短名>`

计划修复接续：`<plan>-plan-<execute/review>；计划书：《<计划书路径>》；任务目标：修复：<最小阻断项>；任务链记录：<记录文件短名>`

## 失败示例

| 失败示例 | 为什么不合格 | 应如何改 |
| --- | --- | --- |
| `demo-S2-实现；计划书：《ARCHITECTURE/plan/demo.md》；任务目标：补齐示例；任务链记录：20260415-demo.md` | 把计划内小任务编号当作独立任务编号，仍是旧小任务流 | 改成计划级任务，例如 `demo-plan-execute；计划书：《ARCHITECTURE/plan/demo.md》；任务目标：按计划书完成 demo 公开 API、实现、测试与计划内 S1/S2 验收；任务链记录：20260415-demo.md` |
| `demo-plan-execute；计划书：《ARCHITECTURE/plan/demo.md》；任务目标：顺手把后面所有内容一起做完；任务链记录：20260415-demo.md` | `任务目标` 扩成长期规划，审查人无法判断当前计划闭环是否完成 | 只写整份计划的直接产出，例如“按计划书完成 xx 公开 API、实现、测试与 pytest -q test/... 验收” |
| `demo-plan-execute；计划书：《ARCHITECTURE/plan/demo.md》；任务目标：补齐示例；任务链记录：20260415-demo.md；worktree=wt-demo；验收标准=全部通过` | 把 `worktree`、验收说明塞进任务描述，和标准短格式混写 | `worktree` 放在任务分发消息里，验收标准放计划书或记录文件里 |
| `实现；任务目标：补齐示例` | 缺少 `任务链记录`，后续无法确定沿用哪份日志 | 补成 `实现；任务目标：补齐示例；任务链记录：20260415-demo.md` |
| `demo-plan-execute；任务目标：补齐示例；任务链记录：20260415-demo.md` | 明明有计划书却漏写 `计划书：《...》`，会导致执行边界不清 | 有计划书时必须显式写出完整计划书路径 |
| `demo-plan-execute；计划书：《ARCHITECTURE/plan/demo.md》；任务目标：继续推进；任务链记录：20260415-demo.md` | `任务目标` 不是可执行动作，执行人无法据此落地实现、测试和验收 | 改成“添加 xxx 接口，并跑通 pytest -q test/... ”或“修复 xxx 失败，使 python3 -m expectation... 通过” |
