# 协作沟通原则

## 文件说明
- 功能说明：统一计划迭代审阅、返工交流、待确认项和跨角色回报的质量标准。
- 使用示例：计划书成稿、任务返工、审查不通过、计划书入档验收复核或权限阻塞时，按本文件组织交流内容。

## 基本原则
- 交流目标是提高质量，不是减少返工；只要能提升可读性、可维护性、测试有效性或边界完整性，就可以提出返工。
- 普通质量问题不暂停任务，直接形成可执行返工项并回到对应角色处理。
- 只有执行链无权处理的问题才暂停，例如公开 `API` 缺用户确认、`expectation/` 本体疑似有问题、权限冲突或合同真源冲突。
- 结论必须可追溯；不得只在口头、会话或 `talk.log` 中保留关键判断。
- 对外消息必须包含任务号或计划路径、对象、证据、最小动作、验收或下一步；不得只写“继续推进”“按当前口径收口”。

## 计划迭代审阅
- 计划书成稿前必须完成 Codex `subagent` 严格审阅持续收敛；每一轮必须基于上一轮修订后的最新计划文本。
- 每轮必须携带标准包，并写清审阅对象、记录路径、严格通过口径、问题、证据、影响、主线处理、状态和遗留项。
- 所有已发起或计划要求的 `subagent` 审阅任务均无阻断、无最小需改项、无待确认项后，计划负责人必须写明 `subagent` 收敛结论，才可请求守护最终检验。
- `subagent` 审阅结论只能作为候选方案、风险和证据；凡涉及公开 `API`、任务范围、依赖顺序、验收口径或 `expectation/` 授权不确定，必须形成待确认项交用户或架构师裁定。
- 公开 `API`、验收资产、任务范围或 `expectation` 授权变化时必须补新一轮严格审阅；仅文字整理可复用原 `迭代审阅记录`，并注明未改变决策。
- 守护最终检验通过前不得请求管理员下发计划；其它计划意见只作审阅输入，不替代守护最终检验。

## 返工交流
- 返工项必须写成 `问题 -> 影响 -> 最小返工动作 -> 验收方式`，并绑定当前 diff、当前计划完成态或可定位文件 / 行。
- 可读性、可维护性、结构清晰度、注释准确性、测试有效性和记录真实性都可以成为返工理由。
- 返工项必须能由接手角色直接执行；不能只写“优化一下”“再看看”“按审查意见处理”。
- 同一任务第二轮及以后返工必须标注 `新增问题 / 重复问题 / 范围扩大`；范围扩大到公开 `API`、`expectation/` 本体或计划目标外时，转架构或用户确认。
- 若返工需要新增公开 `API`、改稳定错误语义或改 `expectation/` 本体，先暂停并转为待确认项。

## 提问与确认
- 正式待确认项向用户或架构师提出时，必须带选项、影响和推荐项。
- 普通状态同步、事实澄清或阻塞回报可用短口径，但必须写清任务号或计划路径、证据、问题和下一步。
- 向管理员提问时，必须写清任务号、记录路径、当前状态和卡住的流程点。
- 向执行人或审查人返工时，必须写清最小返工动作和验收方式。
- 接收方回复后，发起方负责把结论落到计划书、专题 `spec`、任务记录或 `done_plan` 的正确位置。

## 自检
- 本次交流是否有明确问题、证据、影响、责任角色和下一步。
- 是否把普通质量问题直接转成返工，而不是误判为暂停。
- 是否把执行链无权处理的问题转为待确认项。
- 是否已把关键结论写入正式记录。
