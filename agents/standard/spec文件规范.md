# spec 文件规范

## 文件说明

- 功能说明：统一 `spec/` 文档结构、公开 `API` 列表、实现文件定义方式、`API/SPI` 管理、文字写法和测试映射。
- 使用示例：新增或修改公开行为前，先按本文件补齐对应 `spec`；执行人只能按 `spec` 中已定义且已确认的公开 `API` 落地。

## 结构目录

- `功能简介`：简要说明该 `spec` 对应类、模块、方法、工具入口或协议的作用与职责。
- `API 列表`：必须紧跟在 `功能简介` 后，只用于快速列举当前 `spec` 覆盖的公开 `API`。
- `文档信息`：列出 `spec`、功能实现和 `test`，不写人员元信息。
- `依赖`：说明该 `spec` 直接依赖的文件、类型、模块或外部能力，并附对应路径。
- `术语`：可选章节，用于解释本文后续会反复使用且容易混淆的名词。
- `目标`：可选章节，用于说明该模块对外提供的能力、使用方式和预期场景。
- `额外补充`：可选章节，只写模块级、文件级或专题级补充信息；不得承载单个接口的参数限制、错误语义、兼容要求或非目标。
- `API详细说明`：逐个说明公开 `API` 的参数、返回值、使用示例、功能说明和注意事项。
- `测试`：列出测试文件、执行命令、测试目标、功能与用例清单。

## 章节顺序

- `功能简介` 必须是第一个业务章节。
- `API 列表` 必须紧跟 `功能简介`，不得后移到 `文档信息`、`依赖` 或详细说明之后。
- `文档信息` 必须位于 `API 列表` 之后。
- `依赖` 必须位于 `文档信息` 之后。
- `API详细说明` 必须位于可选说明章节之后、`测试` 之前。
- `测试` 必须作为末尾验收章节。
- 除 `术语`、`目标`、`额外补充` 外，不得随意增加“历史记录”“迁移过程”“临时方案”“任务过程”类章节。

## API 列表

- `API 列表` 只做快速索引，不展开行为细节。
- 每个公开 `API` 必须写出完整参数签名。
- 函数签名写成 `api1() -> None` 或 `api2(target: FooTarget, val: int) -> Result`。
- 类构造签名写成 `class Foo(target: FooTarget, val: int)`。
- 类公开方法签名写成 `Foo.run(target: FooTarget, val: int) -> Result` 或 `Foo.stop() -> None`。
- 工具入口必须写命令参数、默认值、输出格式和失败语义。
- include API 必须写命名空间、函数签名、宏名、模板参数和稳定输出。
- 公开错误文本、返回值结构、参数顺序、默认值、工具参数和稳定输出都属于公开 `API`。
- 若该 `spec` 对应 `class`，必须列出该 `class` 的公开 `API` 列表；不在列表中的方法默认视为非公开接口。
- 未满足仓库根目录 `AGENTS.md` 公开 `API` 变更约束的新增、删除、重命名或修改项，不得进入 `API 列表`。

## 文档信息

- `spec`：写当前 `spec` 文件路径。
- `功能实现`：写承载公开行为的实现文件路径；一对多实现必须说明主入口与辅助实现关系。
- `test`：写验证公开行为的测试文件路径；多测试文件必须说明各自覆盖目标。
- 文件路径必须使用仓库相对路径，除用户明确要求外不得写本机绝对路径。
- `expectation` 可在 `测试` 或合同验收中单列为验收资产，不得写成普通 `test` 替代项。
- 人员元信息不是必填结构；不得因为缺少人员元信息退回 `spec`。

## 依赖

- `依赖` 只列公开依赖、公开类型、公开模块或外部工具。
- 不得要求实现或测试跨文件调用非公开 helper、下划线函数、隐藏常量、私有方法或临时兼容 shim。
- 如果公开行为必须依赖新的跨文件 helper，必须先判断它是否应成为公开 `API`；未满足仓库根目录 `AGENTS.md` 公开 `API` 变更约束前不得在 `spec` 中写成已生效接口。
- 一对多实现可以存在，但必须服务同一公开语义，并在 `文档信息` 或 `依赖` 中说明对应关系。
- 多测试文件可以存在，但必须共同锁定同一公开合同，不得顺带挂入其他专题测试。

## API/SPI 管理

- `API` 是用户、测试、工具入口、include 消费者或其他模块可直接依赖的稳定公开接口。
- `SPI` 是为扩展、注册、调度或插件化提供的稳定服务接口；只要允许跨文件或外部消费，也必须按公开 `API` 管理。
- 非公开 helper 只能服务当前文件或当前实现内部，不得在 `spec` 中写成可跨文件调用的能力。
- 兼容别名、旧导入路径、临时桥接层、测试夹具和脚本内部常量不得因为当前可用就写入公开 `API/SPI`。
- 若 `spec` 需要保留兼容入口，必须写清兼容期限、公开行为、禁止扩展范围和确认依据。

## API详细说明

- 每个公开 `API` 必须有对应详细说明条目。
- 每个条目必须包含 `api`、`参数`、`返回值`、`使用示例`、`功能说明` 和 `注意事项`。
- `api` 字段必须与 `API 列表` 中的签名一致。
- `spec` 用来定义实现文件公开行为；实现文件必须按 `spec` 的参数、返回值、错误语义和 API `注意事项` 落地，`spec` 不得把公开语义转交给实现文件。
- `参数` 必须逐个参数单列说明，不得只写“按 API 列表传入”。
- 每个参数必须写清参数含义、类型、取值范围、默认值、是否允许 `None`、是否可变和错误语义。
- 函数签名不得使用泛化的 `object` 表示多种输入；存在多种输入时必须用明确联合类型或多个具体类型写清。
- `返回值` 必须写清类型、结构、稳定字段和异常路径。
- `使用示例` 必须写出最小可读代码块；Python `API` 必须使用 `python` 代码块直接调用当前公开 `API`，不得写成“见测试章节”“参考用例”或只给文字描述。
- `使用示例` 不得调用非公开 helper 或测试专用入口；CLI、include 或脚本入口应使用对应语言或命令代码块展示公开入口。
- `功能说明` 必须描述当前公开行为，不写历史迁移过程或已废弃实现。
- `注意事项` 必须写清该接口自己的边界条件、非目标、性能或资源限制、稳定错误文本和兼容性要求。
- 单个接口的限制不得统一写入 `额外补充` 或旧式边界章节；如果限制只影响某个 API，必须写回该 API 条目的 `注意事项`。

## 文字规范

- 文字必须描述当前合同，不写“之前”“临时”“历史上”“旧方案仍可”等过程性叙述。
- 不得使用“等”“相关”“一类”“若干”“适配一下”“保持一致”这类无法执行或验收的泛化表述。
- 对输入、输出、错误、边界和非目标必须写成可测试语句。
- 公开错误文本必须写成稳定文本；复用错误模板应指向公开错误模块或明确的公开常量。
- 文件路径、命令、类型名、函数名和参数名必须使用反引号。
- 示例必须最小化，只展示公开行为，不混入内部实现细节。
- `spec` 不绑定某一次任务记录或某一轮 review 过程；任务过程写入任务记录，不写入最终 `spec`。

## 测试

- 测试文件必须验证公开 `API`；不得直连跨文件非公开 helper、隐藏常量、私有方法或未定义公开入口。
- 每个公开行为至少对应一个 pytest、测试脚本或可运行本地脚本。
- `expectation` 只能作为合同验收资产单列，不计入 Diff 反推测试，也不能替代改动文件对应测试。
- 稳定错误文本、CLI 参数、include 输出、IR 输出、源码输出和兼容入口都必须有明确验收方式。
- 测试章节必须写清测试文件、执行命令、测试目标、功能与用例清单。
- `测试目标` 必须使用 `### 测试目标` 小节，逐条说明该测试集合验证的公开行为。
- `功能与用例清单` 必须使用 `### 功能与用例清单` 小节，并采用固定表格列：`用例 ID / 功能 / 场景 / 前置条件 / 操作 / 预期结果 / 建议测试`。
- 每个用例行必须写清该 case 测什么、输入或前置条件是什么、执行哪个公开测试入口、预期通过或失败语义是什么；不得只写“覆盖相关行为”“保持一致”这类泛化表述。

## 最小模板

```md
# <spec 标题>

## 功能简介

- `<模块或 API>` 提供 `<公开能力>`，负责 `<边界>`。

## API 列表

- `api1() -> None`
- `api2(target: FooTarget, val: int) -> Result`
- `class Foo(target: FooTarget, val: int)`
- `Foo.run(target: FooTarget, val: int) -> Result`
- `Foo.stop() -> None`

## 文档信息

- `spec`：`<spec_file>`
- `功能实现`：`<impl_file>`
- `test`：`<test_file>`

## 依赖

- `<path>`：`<公开依赖说明>`

## API详细说明

### `api2(target: FooTarget, val: int) -> Result`

- api：`api2(target: FooTarget, val: int) -> Result`
- 参数：
  - `target`：目标对象；类型 `FooTarget`；无默认值，调用方必须显式提供；不允许 `None`；非法目标必须抛出稳定错误。
  - `val`：写入或计算用的数值；类型 `int`；无默认值，调用方必须显式提供；不允许 `None`；越界值必须抛出稳定错误。
- 返回值：`Result`；返回结构、稳定字段和异常路径必须在本文写清。
- 使用示例：

  ```python
  result = api2(target=target, val=1)
  ```
- 功能说明：`<当前公开行为>`
- 注意事项：`<本接口自己的边界、错误、非目标、性能、资源或兼容性限制>`

## 测试

- 测试文件：`<test_file>`
- 执行命令：`pytest -q <test_file>`

### 测试目标

- `<该测试集合验证的公开行为>`

### 功能与用例清单

| 用例 ID | 功能 | 场景 | 前置条件 | 操作 | 预期结果 | 建议测试 |
| --- | --- | --- | --- | --- | --- | --- |
| TC-XXX-001 | `<功能点>` | `<case 场景>` | `<公开输入或夹具条件>` | `<执行 pytest/test script>` | `<稳定通过或失败语义>` | `<test_name>` |
```
