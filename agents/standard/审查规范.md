# 审查规范

## 1. 结论与证据

- 审查结论仅允许：`通过`、`需修改`、`阻塞`。
- 结论必须可复现：给出文件位置、复测命令、关键输出与退出码。
- 仅口头判断、无证据结论无效。

## 2. 通过门槛（全部必选）

- 一致性：`spec`、实现、测试无冲突。
- 正确性：核心行为正确，关键路径可执行。
- 安全性：完成漏洞排查，至少覆盖以下 6 类风险：
  - 输入校验绕过
  - 类型/形状绕过
  - 边界越界
  - 错误处理缺失
  - 状态污染
  - 资源释放问题
- 可维护性：无明显职责混杂/高重复/可读性风险；若发现隐患，必须提出改进建议。
- 可验证性：提供可直接执行的复测命令与结果。
- 注释一致性：涉及函数新增/修改时，必须补齐**中文**注释（至少包含功能说明与使用示例），且与实现一致；审查任务必须核对注释/示例一致性，缺失或不一致判定为不通过。
- 建议闭环：审查中一旦提出“改进建议”，该建议必须转为后续修复任务并完成复核；在建议未落实前不得判定 `通过`。

说明：存在必须修改项、未完成漏洞排查、存在未落实建议、或证据不足时，不得判定 `通过`。

## 3. 检查清单

### 3.1 spec 检查

- 结构符合 `spec 文件规范`，接口描述完整（功能、参数、返回、限制、异常）。
- 测试目标与可执行测试可映射，禁止空泛条目。

### 3.2 实现检查

- 公开语义与 `spec` 对齐，不得隐式改行为。
- 关键校验与防护清晰可见。
- 发现 `spec` 缺失或歧义时，必须标记并给出处置建议。

### 3.3 测试检查

- 覆盖正确性、边界、异常、回归路径。
- 断言可诊断，失败可定位。
- 缺失测试必须明确归类为“必须修改项”或“改进建议”。

### 3.4 注释与示例检查

- 所有新增/修改函数必须有清晰**中文**函数注释，且至少包含功能说明与使用示例；若缺失，必须判定为“需修改”。
- 函数注释中的功能说明必须与实现一致；若存在差异，必须判定为“需修改”。
- 函数注释中的使用示例必须与当前实现一致；示例不一致或缺失，必须判定为“需修改”。

## 4. 问题分级与结论规则

- `P0`：安全漏洞、数据错误、崩溃、严重行为偏差。
- `P1`：接口/语义不一致、关键边界缺失、核心测试缺失。
- `P2`：可维护性与非关键测试改进项。
- 出现 `P0/P1`：结论只能为 `需修改` 或 `阻塞`。
- 若存在任意“改进建议”（含 `P2`），结论不得为 `通过`；需先进入修复任务并复审通过后方可判定通过。

## 5. 审查报告模板（强制）

- `问题列表`：每项包含 `文件/接口`、`现象`、`风险`、`建议`、`优先级`。
- `漏洞排查结果`：列出检查项与结论。
- `改进建议`：如有建议必须明确落地动作与责任人；若无，明确写“未发现额外改进点”。
- `最终结论`：`通过/需修改/阻塞`，并在非通过时给出下一步任务建议。

## 结构性问题如何转成任务

- 若问题不是“补一行代码”能收口，而是涉及范围重排、边界重定、文档/实现/测试三方重新对齐，应视为结构性问题，不能只留口头建议。
- 结构性问题一旦发现，审查记录里必须补两部分：
  - 为什么这是结构性问题。
  - 下一步应转成什么任务。

## 转成任务模板

```text
问题：<结构性问题摘要>
影响：<为什么当前不能直接判通过>
建议动作：转成独立任务继续收口
建议任务类型：<spec/build/review>
建议任务目标：<一句话直接产出>
建议可改范围：<文件或模块范围>
建议验收：<最小命令或资产>
```

## 结构性问题示例

```text
问题：计划书中的公开 API 与当前测试入口命名不一致，执行人无法直接按计划收口
影响：继续推进会造成 spec / 测试 / 任务描述三口径并存
建议动作：转成独立 spec 任务继续收口
建议任务类型：spec
建议任务目标：统一公开 API 命名与计划书示例
建议可改范围：ARCHITECTURE/plan/...、spec/...、对应测试入口文档
建议验收：rg -n "公开 API|示例" <plan/spec> && pytest -q <相关测试>
```
