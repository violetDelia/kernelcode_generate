# 审查规范

## 1. 结论与证据

- 审查结论仅允许：`通过`、`需修改`、`阻塞`。
- 结论必须可复现：给出文件位置、复测命令、关键输出与退出码。
- 仅口头判断、无证据结论无效。

## 2. 通过门槛（全部必选）

- 一致性：`spec`、实现、测试无冲突。
- 正确性：核心行为正确，关键路径可执行。
- 安全性：完成漏洞排查，至少覆盖以下 6 类风险：
  - 输入校验绕过
  - 类型/形状绕过
  - 边界越界
  - 错误处理缺失
  - 状态污染
  - 资源释放问题
- 可维护性：无明显职责混杂/高重复/可读性风险；若发现隐患，必须提出改进建议。
- 可验证性：提供可直接执行的复测命令与结果。
- 执行前提：执行人记录已写清 `执行前阅读记录`、`最小功能闭环` 与 `自检`；缺失且无有效说明时不得通过。
- Diff 反推覆盖：审查必须读取实际改动文件清单，按 diff 反推对应测试；不得只沿用任务树或计划书写出的验收命令，且不得把 `expectation` 计入 diff 反推测试。
- 注释一致性：涉及函数新增/修改时，必须按 [`实现文件规范.md`](实现文件规范.md) 补齐**中文**注释（至少包含 `创建者 / 最后一次更改 / 功能说明 / 使用示例`），且与实现一致；审查任务必须核对注释/示例一致性，缺失或不一致判定为不通过。
- 建议闭环：审查中一旦提出“改进建议”，该建议必须转为后续修复任务并完成复核；在建议未落实前不得判定 `通过`。
- 严格改进门槛：只要还能明确指出任何一处可执行的改进点、收敛点、可维护性优化点或测试增强点，就不得判定 `通过`；必须写成 `需修改`，并把这些点列入问题列表或改进建议。
- 合同判断一致性：若 `expectation` 路径正确、入口可执行、输出明确，则默认按该合同审查实现与测试；不得把“实现未满足既有 expectation”误判成需要架构师再次裁决的问题。

说明：存在必须修改项、未完成漏洞排查、存在未落实建议、或证据不足时，不得判定 `通过`。

## 3. 检查清单

### 3.1 spec 检查

- 结构符合 `spec 文件规范`，接口描述完整（功能、参数、返回、限制、异常）。
- 接口设计合理，无明显职责重叠、功能冗余、重复定义或难维护的扩展口。
- 当前 `spec` 必须明确列出本轮允许实现的 `API` 集合；若 `spec` 允许执行人自行补 helper、隐藏接口或未命名函数，视为边界不清。
- `API 列表` 必须紧跟在“功能简介”后，且只做快速索引；缺失、后移或把这一节写成大段接口正文，都视为不通过。
- 每个公开 `API` 都必须写出参数签名；若是 `class` 场景，必须逐条列出该类的公开 `API`，未列出的接口默认不属于公开面。
- 审查 `spec` 时必须核对：哪些是当前允许实现的公开 `API`，哪些只能作为当前文件内辅助函数存在；若正文没有把这条边界讲清，视为 `spec` 不通过。
- 功能边界、异常情况、非目标和兼容口径清楚，下游执行人不需要猜。
- 若本轮关联实现或测试已经存在，审查需核对是否出现未在 `spec` 中定义的非 `API` 函数、是否存在跨文件使用非公开 `API` 的行为，以及测试是否直接调用非 `API` 接口；若存在，必须在 `spec` 审查里指出。
- 若修改公开 `API`、IR 文本、脚本入口或错误语义，必须检查兼容性、破坏性变更、弃用路径和旧调用残留。
- 需要关注性能、可靠性、安全、并发或资源生命周期的接口，必须写明预期行为和失败处理。
- 文字能一眼看懂，术语、参数顺序、示例和输出无歧义。
- 测试目标与可执行测试可映射，禁止空泛条目。

### 3.2 实现检查

- 公开语义与 `spec` 对齐，不得隐式改行为。
- 允许在当前文件内新增为实现当前 `spec` 公开 `API` 服务的辅助函数；但不得新增未在当前 `spec` 明确定义的公开接口，也不得以直接调用、包装转发、别名导入、反射或其他方式使用当前文件之外的非公开 `API` 函数、方法或隐藏 helper；若 diff 出现这类越界新增或调用，默认判定为 `需修改`。
- 审查人不得以“只是内部 helper”“测试方便”“当前能跑”为由放行任何当前文件之外的非公开 `API` 使用。
- 若本轮改动涉及功能实现文件，审查必须核对该文件的文件级说明是否已按 [`实现文件规范.md`](实现文件规范.md) 补齐或更新 `API 列表`；`API 列表` 应紧跟在 `功能说明` 后，只做快速索引，列出当前文件公开 `API` 与参数签名。若是 `class` 文件，必须逐条列出类公开 `API`；缺失、后移或与实现不一致时，结论不得通过。
- 关键校验与防护清晰可见。
- 实现无明显遗漏、重复代码、职责混杂、过大函数、不合理的一两行小函数，且可维护性和扩展性足够。
- 新增或修改函数的注释、示例和行为保持一致，不得出现过时注释或误导性示例。
- 可复用逻辑应复用或合理抽取；若选择不抽取，审查记录应说明原因。
- 输入/输出校验、错误处理、资源释放、并发安全、性能影响和向后兼容性应与任务风险匹配；缺失时不得只以“当前 case 能跑”放过。
- 发现 `spec` 缺失或歧义时，必须标记并给出处置建议。
- 若 `expectation` 已被计划书或规则指定为合同真源，审查应优先检查实现是否满足该 `expectation`，而不是建议执行人回退去请求架构师再次解释合同。

### 3.3 测试检查

- 覆盖正确性、边界、异常、回归路径。
- 断言可诊断，失败可定位。
- 测试只能验证当前 `spec` 定义的公开 `API`、公开入口或明确点名的稳定内部契约；若测试直接调用非 `API` 函数、隐藏 helper、未在 `spec` 定义的接口，或借助跨文件非公开 `API` 绕过公开行为，默认判定为 `需修改`。
- 缺失测试必须明确归类为“必须修改项”或“改进建议”。
- 审查人必须基于实际 diff 逐项反推测试：每个改动文件都要能对应到已执行测试、审查复跑命令或明确的未覆盖原因。
- 审查必须主动找特殊情况、边界遗漏、内容不完整、冗余、维护性、扩展性和可读性问题；每条测试应确认断言有效且能定位失败；任何不完美但可修改的点都要写成问题、建议或后续任务。
- 人写代码原则上要逐行理解；若只审了部分文件或某个方面，审查记录必须写清范围。遇到安全、并发、性能、隐私、架构或领域知识不足时，必须要求领域专家补审。
- 若执行人日志缺少 `执行前阅读记录`、`最小功能闭环` 或 `自检`，或自检只写“已自检”“无问题”等空泛结论，且无法从记录中确认其已读阶段正文、边界、闭环目标和角色重点检查结论，结论必须为 `需修改`。
- 若执行人日志缺少 `Diff 反推自测`，或审查人无法从记录中确认改动文件对应测试已覆盖，结论必须为 `需修改`，问题写成“测试覆盖不足”。
- 若 diff 反推出的测试失败，只有在记录了本轮前基线、确认与本轮无关、且有架构师明确认可时，才可不作为当前阻断。

### 3.4 注释与示例检查

- 所有新增/修改函数必须有清晰**中文**函数注释，且至少包含 `创建者 / 最后一次更改 / 功能说明 / 使用示例`；若缺失，必须判定为“需修改”。
- 所有新增/修改的功能实现文件都必须有文件级说明，且至少包含 `创建者 / 最后一次更改 / 功能说明 / API 列表 / 使用示例 / 关联文件`；若缺失，必须判定为“需修改”。
- 文件级 `API 列表` 必须紧跟在 `功能说明` 后，并列出完整参数签名；若缺失、后移、漏签名或混入私有 helper，必须判定为“需修改”。
- 函数注释中的功能说明必须与实现一致；若存在差异，必须判定为“需修改”。
- 函数注释中的使用示例必须与当前实现一致；示例不一致或缺失，必须判定为“需修改”。

## 4. 问题分级与结论规则

- `P0`：安全漏洞、数据错误、崩溃、严重行为偏差。
- `P1`：接口/语义不一致、关键边界缺失、核心测试缺失。
- `P2`：可维护性与非关键测试改进项。
- “实现未满足既有 expectation 合同”默认按 `P1` 的实现问题处理，不按“阻塞等待架构师”处理；只有 `expectation` 本身存在路径、入口或合同冲突时，才允许升级为 `阻塞`。
- 出现 `P0/P1`：结论只能为 `需修改` 或 `阻塞`。
- 若存在任意“改进建议”（含 `P2`），结论不得为 `通过`；需先进入修复任务并复审通过后方可判定通过。

## 5. 审查报告模板（强制）

- `问题列表`：每项包含 `文件/接口`、`现象`、`风险`、`建议`、`优先级`。
- `漏洞排查结果`：列出检查项与结论。
- `执行前提核对`：说明是否已核对 `执行前阅读记录`、`最小功能闭环` 与 `自检`。
- `Diff 反推审查`：列出被审 diff 文件、反推出的测试命令、审查复跑或核验证据、未覆盖项；若另跑 `expectation`，单列为合同验收。
- `改进建议`：如有建议必须明确落地动作与责任人；若无，明确写“未发现额外改进点”。
- `最终结论`：`通过/需修改/阻塞`，并在非通过时给出下一步任务建议。
- 若本次审查属于计划书终验、复验或终验修复复核：
  - 给出结论的架构师必须把检验结论写回计划书正文，至少写清 `结论人 / 结论 / 验证基线 / 最小阻断项或通过摘要`。
  - 若结论为非通过态，需先把最小阻断项整理成明确修复范围，再继续推进修复任务。
  - 管理员若尚未看到修复任务，统一询问 `守护最好的爱莉希雅` 补建；另一位架构师只补充阻断项、修复边界和验收重点，不重复创建同一修复任务。

## 结构性问题如何转成任务

- 若问题不是“补一行代码”能收口，而是涉及范围重排、边界重定、文档/实现/测试三方重新对齐，应视为结构性问题，不能只留口头建议。
- 结构性问题一旦发现，审查记录里必须补两部分：
  - 为什么这是结构性问题。
  - 下一步应转成什么任务。

## 转成任务模板

```text
问题：<结构性问题摘要>
影响：<为什么当前不能直接判通过>
建议动作：转成独立任务继续收口
建议任务类型：<spec/build/review>
建议任务目标：<一句话直接产出>
建议可改范围：<文件或模块范围>
建议验收：<最小命令或资产>
```

## 结构性问题示例

```text
问题：计划书中的公开 API 与当前测试入口命名不一致，执行人无法直接按计划收口
影响：继续推进会造成 spec / 测试 / 任务描述三口径并存
建议动作：转成独立 spec 任务继续收口
建议任务类型：spec
建议任务目标：统一公开 API 命名与计划书示例
建议可改范围：ARCHITECTURE/plan/...、spec/...、对应测试入口文档
建议验收：rg -n "公开 API|示例" <plan/spec> && pytest -q <相关测试>
```
