# 审查规范

## 文件说明
- 功能说明：统一 review / 复审 / 终验修复复核的结论、证据和最小需改项写法。
- 使用示例：审查任务开始前，按本文件确定必须检查的内容。

## 审查重点
- 优先找 bug、行为回归、边界遗漏、异常路径、资源问题、并发问题、性能风险和测试无效。
- 核对公开 `API` 变更是否有用户确认来源。
- 核对实现是否跨文件调用非公开 `API`。
- 核对测试是否直连非 `API` helper。
- 核对任务记录是否包含执行前阅读、最小功能闭环、自检和 diff 反推自测。
- 核对文件级说明、`API 列表`、函数注释、示例和 `spec/test` 是否同步。
- 核对函数签名是否用明确类型表达输入；不得用 `object` 掩盖可枚举输入。
- 核对实现是否存在跨文件非公开 `API` 使用、上下文运行时能力探测或非装饰器场景的嵌套函数。

## 结论
- `通过`：无可执行改进项，验证和记录完整。
- `最小需改项`：存在 1 到 3 个可执行阻断或高价值收口点。
- `不通过`：方向、边界、实现或验证不满足当前任务目标。
- 只要还有明确可改进点，不得写 `通过`。
- 不确定是否影响公开行为时，先写成问题或待确认，不用乐观放行。

## 证据要求
- findings 必须排在前面，按严重程度排序。
- 每条问题写清文件、行号或可定位位置、影响和最小修复建议。
- 验证命令写清命令、退出码、通过数量或失败摘要。
- 未执行测试时写明原因和残余风险。
- `expectation` 结果单列为合同验收，不计入 diff 反推审查。

## 必查项
- 执行人是否只改任务允许范围。
- 公开 `API` 变更是否有用户确认来源。
- 实现是否只跨文件调用公开 `API`。
- 测试是否只验证公开 `API`。
- diff 反推测试是否覆盖实际改动。
- 失败项是否有基线证据和合理判断。
- 是否存在遗漏、重复、过时注释、过大函数、无意义小函数、未释放资源或不可维护结构。
- 是否错误地把人员元信息当成文件级说明、函数注释或 `spec` 的强制项。

## 不得放行
- 公开 `API` 变更缺用户确认来源。
- 实现跨文件调用非公开 helper。
- 测试绕过公开 `API` 直连内部 helper。
- 函数签名用 `object` 掩盖可枚举输入类型。
- 使用 `hasattr/getattr/callable(getattr(...))` 等上下文能力探测绕过公开 `API`。
- 非装饰器场景在函数体内定义嵌套函数。
- `expectation/` 改动缺用户或架构师授权。
- 执行记录缺关键验证且无有效说明。
- 有可执行改进项但仅写“建议后续优化”。

## 记录模板
```md
结论：通过 / 最小需改项 / 不通过
验证基线：<commit / worktree / 执行目录>
发现：
- <severity> <file:line> <问题 + 影响 + 建议>
验证：
- <command> -> <result>
残余风险：<如无写无>
```
