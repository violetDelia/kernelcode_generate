# 审查规范

## 文件说明
- 功能说明：统一 review / 复审 / 计划书入档验收修复复核的结论、证据和最小需改项写法。
- 使用示例：审查任务开始前，按本文件确定必须检查的内容。

## 审查重点
- 优先找 bug、行为回归、边界遗漏、异常路径、资源问题、并发问题、性能风险和测试无效。
- 可读性、可维护性、结构清晰度、注释准确性、函数粒度和测试断言有效性都属于审查重点；即使不完全属于原任务目标，也可以要求返工。
- 审查必须判断当前 diff 是否增加理解成本或修改成本；新增分支、兼容层、抽象、状态或依赖如果没有清晰收益，应要求收敛。
- 核对模块 / helper 是否足够“深”：对外接口简单明确，内部隐藏实现细节；只转发、只改名、只包一层的浅抽象应删除或合并。
- 核对命名是否表达领域意图、单位、边界、错误含义和生命周期；含糊名称或需要读实现才能理解的名称应返工。
- 核对重复是否属于同一知识重复；同一规则、常量、错误语义或转换逻辑多处散落时必须收敛到单一真源。
- 核对注释和文档是否解释原因、约束、不变量和取舍；复述代码、过期迁移说明或掩盖坏设计的注释不得放行。
- 核对公开 `API` 变更是否有用户确认来源。
- 核对实现是否跨文件调用非公开 `API`。
- 核对测试是否直连非 `API` helper。
- 核对任务记录是否包含执行前阅读、最小功能闭环、自检和 diff 反推自测。
- 核对文件级说明、`API 列表`、函数注释、示例和 `spec/test` 是否同步。
- 核对函数签名是否用明确类型表达输入；不得用 `object` 掩盖可枚举输入。
- 核对实现是否存在跨文件非公开 `API` 使用、上下文运行时能力探测或非装饰器场景的嵌套函数。

## 减法审查
- 审查必须回答：本轮新增或改动逻辑替代了什么旧逻辑、旧 helper、旧入口、旧测试或旧文案。
- 审查必须回答：被替代旧逻辑是否已经删除；未删除时，执行记录是否写清保留依据、风险和验证命令。
- 审查范围限定为本轮 diff、diff 直接触达模块、执行记录声称替代的旧路径；全仓扫描只作线索，非本轮直接相关存量不作为当前阻断。
- 对执行记录声称删除或替代的旧路径，审查应使用 `rg`、diff 或测试证据核对删除 / 保留结论。
- 只加新实现、不删旧实现，且无保留依据时，不得放行；必须退回 execute 写清删了什么、保留什么、为什么保留。

## 私有函数审查
- 审查必须检查本轮新增或改动的 `private callable`，包括 module-level `_helper`、class `_method`、私有 `staticmethod` 和私有 `classmethod`。
- `__dunder__` 协议方法不按 `private callable` 处理，但仍按公开 / 协议行为审查。
- 新建或改动 `private callable` 小于 5 行有效代码时不得放行；有效代码行不计空行、注释、docstring、函数签名、装饰器行、单独括号行和单独逗号行。
- `private callable` 调用其它 `private callable` 时不得放行；应退回 execute 内联、合并到单一当前文件 helper、提升为已确认公开 API，或重构调用点。
- 测试直连 `private callable`、跨文件调用 `private callable`、通过包装转发或别名导入绕过公开 API，均按公开边界问题退回。

## 审查前置
- `review`、复审和计划书入档验收修复复核给结论前，必须先同步或核对最新主线现场。
- 记录执行目录、目标分支或基线、`fetch` / 对齐或 `merge-base` 核对结果；确认不会覆盖任务 diff、用户改动或其他任务本地改动。
- 同步存在冲突、覆盖风险或无法判断待审 diff 基线时，结论不得继续推进，先暂停并回报管理员或架构师。
- 审查人不得手工改 `TODO.md`、`DONE.md` 或任务表；只允许在角色权限允许时通过标准脚本续接状态，计划级 `review` 通过必须续接或回报管理员进入 `archive_acceptance / 计划书入档验收`，不得直接进入 `merge`。
- 审查计划书流程类改动时，必须核对两轮 `subagent` 严格审阅、标准包、用户待决策项收口和守护最终检验是否仍是下发前硬门禁；普通计划意见不得替代守护最终检验。

## 结论
- `通过`：无可执行返工项，验证和记录完整。
- `最小需改项`：存在 1 到 3 个可执行阻断或高价值收口点，属于不通过的一种，必须回 `execute` 或对应责任角色收口；不得进入 `merge`、不得写计划书入档验收通过。
- `不通过`：方向、边界、实现或验证不满足当前任务目标。
- 只要还有明确可执行返工项，不得写 `通过`；可读性和可维护性返工项同样有效。
- 不确定是否影响公开行为时，先写成问题或待确认，不用乐观放行。

## 证据要求
- findings 必须排在前面，按严重程度排序。
- 每条问题写清文件、行号或可定位位置、影响、最小修复建议和验收方式。
- 返工项必须与当前 diff、当前计划完成态或当前验收资产相关；无法定位、无法验证或收益无法短期证明的内容，写为后续建议或转架构 / 用户确认。
- 审查和计划书入档验收应尽量一次列齐可执行问题，并写清已扫描范围与未覆盖范围。
- 验证命令写清命令、退出码、通过数量或失败摘要。
- 未执行测试时写明原因和残余风险。
- `expectation` 结果单列为合同验收，不计入 diff 反推审查。

## 静态扫描命中分类
- 阻断命中：跨文件非公开 `API` 调用、测试直连非 `API` helper、上下文能力探测、非装饰器嵌套函数、未授权 `expectation/` 改动、公开 `API` 变更缺用户确认。
- 允许命中：定义公开 `API` 本体、同文件内部 helper、装饰器必要闭包、文档示例或历史记录引用。
- 需人工说明命中：`rg` 命中的 `object`、`getattr`、`hasattr`、`def`、脚本参数或路径示例无法机械判断时，必须写清命中位置、判定依据和是否阻断。

## 必查项
- 执行人是否只改任务允许范围。
- 公开 `API` 变更是否有用户确认来源。
- 实现是否只跨文件调用公开 `API`。
- 测试是否只验证公开 `API`。
- diff 反推测试是否覆盖实际改动。
- 失败项是否有基线证据和合理判断。
- 是否存在遗漏、重复知识、过时注释、过大函数、浅模块、无意义小函数、未释放资源或不可维护结构。
- 是否存在可读性差、命名含糊、分支过深、职责混杂、注释与行为不一致或测试断言不能证明行为的问题。
- 是否存在投机泛化、临时兼容层、隐藏状态、跨层耦合、错误语义散落或只为当前 case 服务的实现。
- 是否完成减法审查，能说明旧逻辑是否删除、保留依据是否充分。
- 是否完成私有函数审查，能说明本轮 `private callable` 是否符合 5 行有效代码规则且没有调用其它 `private callable`。
- 是否错误地把人员元信息当成文件级说明、函数注释或 `spec` 的强制项。

## 不得放行
- 公开 `API` 变更缺用户确认来源。
- 新建或重大修订计划缺少两轮 `subagent` 严格审阅、标准包、用户待决策项收口或守护最终检验。
- 实现跨文件调用非公开 helper。
- 测试绕过公开 `API` 直连内部 helper。
- 函数签名用 `object` 掩盖可枚举输入类型。
- 使用 `hasattr/getattr/callable(getattr(...))` 等上下文能力探测绕过公开 `API`。
- 非装饰器场景在函数体内定义嵌套函数。
- `expectation/` 改动缺用户或架构师授权。
- 执行记录缺关键验证且无有效说明。
- 有可执行返工项但仅写“建议后续优化”。
- 仍有可读性、可维护性、测试有效性或边界完整性的可执行返工项。
- 无依据保留被本轮替代的旧 helper、旧入口、旧测试或旧文案。
- 新建或改动 `private callable` 小于 5 行有效代码。
- `private callable` 调用其它 `private callable`。

## 审查自检
- 是否逐项读取实际 diff，而不是只看执行摘要。
- 是否按当前任务和实际 diff 反推测试，并核对测试断言会在实现错误时失败。
- 是否检查了公开 `API`、`expectation/` 权限、跨文件非公开 `API`、文件级说明和函数注释。
- 是否主动查找复杂度增长、浅抽象、重复知识、命名含糊、注释失真、职责混杂和长期维护风险。
- 是否完成减法审查和私有函数审查，并把删了什么、保留什么、为什么保留写入审查结论。
- 是否把每个返工项写成可执行动作，并给出验收方式。

## 记录模板
```md
结论：通过 / 最小需改项 / 不通过
验证基线：<commit / worktree / 执行目录>
发现：
- <severity> <file:line> <问题 + 影响 + 建议>
验证：
- <command> -> <result>
残余风险：<如无写无>
```
