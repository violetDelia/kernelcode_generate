# 协作执行通用规则

## 文件说明
- 功能说明：统一任务执行、记录、续接、API 边界、验收和异常升级口径。
- 使用示例：角色提示词只写角色差异，具体执行约束引用本文件。

## 通用规则
- 所有角色先遵守仓库根目录 `AGENTS.md`，再遵守本文件和个人提示词。
- 当前角色名单、角色名和 session 真源以 `agents/codex-multi-agents/agents-lists.md` 为准；旧同名 session 或备份目录只作历史资料，追溯上下文优先读取 `agents/codex-multi-agents/log/task_records/` 与当前仓库仍保留的任务记录。
- 任务状态通过脚本变更；不得手工改 `TODO.md`、`DONE.md` 或任务表。
- 脚本示例里的 `TODO.md` 是状态文件占位；实际执行时使用管理员下发的状态文件路径。独立 `worktree` 内没有 `TODO.md` 时，使用主仓绝对路径或先问管理员确认，不得猜测路径。
- 状态脚本只负责任务链状态；发起人必须按当前角色权限执行，计划级创建、分发、暂停、归档和最终状态协调由管理员负责。
- 非管理员角色使用 `-next -auto` 只表示按当前任务链续接到下一阶段，不代表本人创建、分发或承接下一阶段；任务类型、接手角色或状态责任不清时，必须先回报管理员，由管理员续接。
- 任何推进动作前必须先补齐任务记录、阻塞记录或评审记录。
- 常规任务记录写入对应任务 `worktree`；无独立 `worktree` 的计划互评、专题 `spec` 互评、终验和归档结论写入计划书、专题 `spec` 或 `done_plan`。
- 咨询、互评、裁定和回报必须可追溯，不用口头结论替代记录。
- 多人讨论和返工交流按 [`协作沟通原则.md`](协作沟通原则.md) 执行，问题必须带证据、影响、最小下一步和责任角色。

## 任务边界
- 一个计划书对应一个计划级 `execute` 任务；计划内 `S1/S2/...` 只是任务卡。
- 已创建的存量旧任务链按现有 `TODO`、任务记录和计划正文继续收口；新建或重大修订计划一律使用一个计划书对应一个计划级 `execute` 任务的新流程。
- 存量任务接续最小补齐：状态文件路径、当前阶段、责任角色、存活记录路径、最新同步现场、未收口返工项、`expectation` scope，以及是否需要补五行短口径。
- `execute` 完成计划内规格、实现、测试与验收闭环后进入 `review`。
- 计划级任务固定流转为 `execute -> review -> 架构复核 / 终验 -> merge/归档`。
- `review` 不通过回到 `execute`；架构复核 / 终验不通过也回到 `execute`，不得另设独立 `refactor` 阶段。
- 非计划级任务可在 `review` 通过且无可执行返工项后按任务链进入 `merge`。
- 任务开始前由架构师草拟或调整计划书和专题 `spec`；`execute` 开始后，任务内 `spec` 改动由执行链负责闭环，除非需要架构裁定或终验修复复核。
- 计划终验只基于当前计划正文、公开 `API`、当前验收资产和最新同步现场。
- 修复接续继续挂在原计划级任务链下，目标直接写清最小阻断项和验收结果。

## 执行前核对
- 核对任务目标是否可执行，是否写清对象、公开行为和验收结果。
- 核对计划书、专题 `spec`、任务记录路径、工作目录、禁止修改面和合同真源。
- 核对是否涉及公开 `API` 变更；若涉及，必须看到用户确认来源。
- 核对是否涉及 `expectation/` 本体；若涉及，只有用户或架构师能处理。

## 最新同步现场
- `review`、复审、架构复核、终验和修复复核给结论前，必须先同步或核对最新主线现场。
- 同步动作至少记录执行目录、目标分支或基线、`fetch` / 对齐或 `merge-base` 核对结果；不得覆盖任务 diff、用户改动或其他任务本地改动。
- 若同步会产生冲突、覆盖风险或无法确认待审 diff 基线，立即暂停并回报管理员或架构师，不得基于过期现场放行。
- 终验和复验还必须记录当前必过合同验收资产的执行目录、环境变量、日志路径和结果摘要。

## API 与实现边界
- 公开 `API` 变更必须先有用户确认来源。
- `execute` 只能实现当前 `spec` 明确定义的公开 `API`。
- 当前文件之外只允许调用对应 `spec` 和文件级 `API 列表` 中声明的公开 `API`。
- 当前文件内可新增服务于公开 `API` 的非公开 helper；不得把 helper 升级为公开入口。
- 测试只能验证公开 `API`；不得直连跨文件非公开 helper。
- 需要新增公开入口、改工具参数或改稳定错误文本时，先暂停并回到用户确认。

## 代码质量原则
- 质量目标是降低后续理解和修改成本；实现应优先减少偶然复杂度，而不是堆叠临时分支、兼容桥或特殊 case。
- 公开接口保持小而明确，内部实现可以承载复杂度；不要为了“看起来分层”拆出浅模块、无意义 helper 或只转发的抽象。
- 命名必须表达领域意图、单位、边界和失败含义；需要读实现才能猜含义的名称应返工。
- 重复逻辑和重复知识必须收敛到单一真源；但不得为了消除少量表面重复抽出更难理解的泛化层。
- 注释和文档解释为什么、公开约束、不可见不变量、错误语义和取舍；不得重复代码表面动作或保留历史过程。
- 重构必须保持公开行为，并由 diff 反推测试或合同验收证明；不可把“整理代码”变成顺手改公开语义。

## 验证
- `execute`、execute 返工、审查和复审必须按实际 diff 反推测试。
- diff 反推测试只包含 `pytest`、测试脚本或可作为测试运行的本地脚本。
- `expectation` 是合同验收资产，不计入 diff 反推测试。
- 任务要求命令是最低集合；实际改动触及更多模块时，必须补充对应测试或写清未运行原因。
- 只有当前计划正文列为必过、且合同真源仍有效的 `expectation` 才驱动实现收口；疑似旧合同、未授权本体问题或与计划不一致时，先记录 actual / expected / spec / verdict 并转架构裁定。
- 终验、复验和终验修复复核必须运行当前计划正文列为必过合同验收资产的相关 `expectation`；若正文已把某入口收为历史 / 本地只读来源说明，或 latest main 明确不存在且不再列为必过入口，不得再把它当作当前阻断项。用户要求全量时才运行全量。
- 验证不能只证明“命令跑过”；必须能说明当前公开行为、失败路径、边界条件和测试断言有效。
- 修改标准、计划书或提示词时，至少执行文本残留扫描、结构检查、当前任务相关引用检查和脚本示例权限一致性扫描。
- 最小验证矩阵：实现改动至少包含 diff 反推 pytest / 脚本、公开 API 与敏感边界静态核对；标准或提示词改动至少包含标题结构检查、旧口径残留扫描、引用路径检查、`.skills` 与 `expectation/` 空 diff 核对；只读评估至少记录读取范围、结论和未修改文件说明。
- 长耗时或易波动验收必须记录固定命令、`cwd`、关键环境变量、timeout、cold / warm cache 说明、日志路径和是否允许重跑；不得用单项通过替代计划要求的 full suite。

## 记录要求
- `execute`、execute 返工和终验修复必须写执行前阅读、最小功能闭环、自检和 `Diff 反推自测`。
- `review`、复审和终验修复复核必须写 `Diff 反推审查`。
- 验证字段必须写命令、退出码、通过数量或失败摘要。
- 已知失败必须写明复现、基线证据、是否本轮引入和当前处理建议。
- 自检必须按角色写，不得只写“已检查”：架构师查计划质量和验收；execute 查实现质量、可读性、可维护性和测试有效性；review 查问题遗漏、边界、测试和返工项；merge 查合入范围和授权；管理员查记录、角色和流转边界；替补按本轮承担角色补齐对应自检。
- 代码类自检必须覆盖：复杂度是否下降或至少未上升、接口是否最小且清晰、命名是否表达意图、重复知识是否收敛、注释是否解释约束而非复述代码、测试是否锁定公开行为。
- full expectation 或 runner 类验收记录必须写 `cwd`、`PYTHONPATH`、`EXPECTATION_WORKTREE_ROOT`、sync worktree、授权 `expectation` scope、runner 特例和日志路径。

## 返工口径
- 审查和终验保持高强度返工；只要能提升当前改动质量、代码可读性、可维护性、测试有效性、边界完整性或实现清晰度，就可以退回返工。
- 返工项写成 `问题 -> 影响 -> 最小返工动作 -> 验收方式`，并绑定当前 diff、当前计划完成态或可定位文件 / 行。
- 审查和终验应尽量一次列齐可执行问题，并写明已扫描范围与未覆盖范围；同一任务第二轮及以后返工必须标注 `新增问题 / 重复问题 / 范围扩大`。
- 同一阻断项重复出现，或进入第三轮返工前，管理员必须转架构师裁定根因，并决定缩小任务、重写最小阻断项、换人或提交用户确认。
- 范围扩大到公开 `API`、`expectation/` 本体或计划目标外时，先由架构师写最小继续边界；需要需求取舍或授权时交用户确认。
- 普通质量问题不暂停任务；直接回到 `execute` 或对应角色处理。
- 只有执行链无权处理的问题才暂停，例如公开 `API` 缺用户确认、`expectation/` 本体疑似有问题、权限冲突或合同真源冲突。

## 异常升级
- 流程、权限、文件范围、任务脚本问题问管理员。
- 实现边界、接口语义、验收口径、架构争议问架构师。
- 需求取舍、公开 `API` 变化和 `expectation` 修改授权问用户。
- 任何角色发现任务目标泛化、授权缺失、记录缺失或验收不可执行，都应先判断是否属于执行链无权处理；无权处理才暂停并写阻塞记录，可直接修复的质量问题转为返工项。
