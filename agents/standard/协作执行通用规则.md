# 协作执行通用规则

## 文件说明
- 功能说明：统一多角色协作时的执行顺序、任务续接、回报方式与管理员推进原则，减少角色提示词中的重复描述。
- 使用示例：角色提示词可直接引用本文件，例如：`通用执行规则以 [协作执行通用规则](../../../standard/协作执行通用规则.md) 为准。`
- 使用示例：执行人完成任务后，可先按本文件确认“记录 -> `-next -auto -type` -> `-talk` 通知管理员”的顺序是否正确。

## 通用执行规则
- 所有角色先遵守仓库根目录 `AGENTS.md`，再遵守本文件与各自提示词。
- 任务状态只能通过脚本变更，不得手工修改 `TODO.md`、`DONE.md` 或任务表内容。
- 所有角色在推进链路前，都必须先补齐任务日志或评审记录，再执行脚本或发消息。
- 若任务有独立 `worktree`，常规任务日志必须写入该 `worktree` 下的对应记录文件；禁止在主仓根目录更新常规任务日志。
- 只有无独立任务 `worktree` 的计划互评、专题 `spec` 互评、终验或归档结论，才按规则写入计划书、专题 `spec` 正文或 `done_plan` 记录文件。
- 咨询问题统一使用 `-talk`；若只是澄清、确认、互评意见同步，不新建任务。
- 流程、权限、文件范围、任务脚本问题问管理员；实现边界、接口语义、验收口径、架构争议问架构师。
- 对 `build` 阶段，若 `expectation` 路径正确、入口可执行、输出明确，则默认视为当前合同真源；实现与其冲突时，执行人应优先修改实现与测试对齐，不因“实现不一致”自动回找架构师。
- 对 `build` 阶段，除 `expectation / spec / 计划书 / agents 标准文档 / immutable 内容` 等显式禁止面外，执行人默认可修改与任务目标直接相关的实现、测试和必要辅助文件；不再要求计划书为实现侧逐文件开白名单。

## 通用任务链路
- 计划书、专题 `spec` 与小任务由架构师先互评，通过后再使用 `-new` 建任务，并通知管理员推进。
- 计划书未完成互评前，不得建任务或通知管理员推进。
- `spec/build/review/替补` 角色完成当前任务后，顺序固定为：写任务日志 -> 执行 `-next -auto -type` -> 用 `-talk` 通知管理员推进。
- 管理员优先推进正在进行中的链路，即优先处理执行者 `-next -auto -type` 后形成的后续任务，再处理新建待分发任务。
- `李白` 只负责合并；顺序固定为：写工作日志 -> 执行合并 -> 对当前 `merge` 任务执行 `-done` -> 用 `-talk` 通知管理员继续推进。
- 计划归档只在两位架构师都明确给出“通过”结论后推进；管理员必须先为该计划创建对应归档任务，再执行 `-done-plan`。
- 归档任务的记录文件统一写入 `agents/codex-multi-agents/log/task_records/done_plan/<年份>/<周>/`；不得只做口头归档，也不得在无任务状态下直接归档。
- 计划书终验、复验或终验修复复核，必须基于“最新同步现场”执行，并在计划书正文或终验记录里明确写出验证基线，例如具体 `worktree`、分支 / `commit`、执行目录或“已同步到最新主线”的说明；不得只写“当前主仓”而不说明基线。
- 若根目录旧主仓工作目录与最新同步现场不一致，只记为现场状态差异；除非相同缺口在已写明的最新同步现场中也复现，否则不得单独将旧工作目录状态视为功能阻断，也不得据此重开实现、`expectation` 或计划修复任务。
- 计划书终验若结论为“不通过”，不得归档；必须先由架构师提炼最小阻断项，再直接创建修复任务，继续把原计划收口到可通过状态。
- 计划书终验若结论为“不通过”，默认由“给出最终不通过结论的那位架构师”创建修复任务；另一位架构师只补充阻断项和修复范围，不再重复建同一任务。
- 修复任务完成后，必须回到原计划书重新验收；只有原计划书重新得到双架构师“通过”结论，才允许通知管理员归档。
- 若两位架构师误建了同一修复任务，只保留一个任务继续推进；保留项以“范围更清楚、依赖正确、worktree 明确”的任务为准，另一项记为重复任务，由管理员停止推进。

## 自动续接口径
- `-next -auto -type` 只用于当前任务已完成、且需要显式创建下游任务的场景。
- `-type` 必须写明下游任务类型；默认推荐链路为：`spec -> build -> review -> merge`。
- 若当前任务未完成、仍有阻塞或仍需澄清，不得使用 `-next -auto -type` 假装续接；应先写阻塞记录并用 `-talk` 回报。
- 若问题仅属于“实现未满足既有 expectation 合同”，当前任务不应以此暂停；应继续保持 `doing`，由执行人直接收口实现。

## 状态流转图

```text
计划书 / 专题 spec 互评通过
  -> 架构师执行 -new
  -> 任务列表
  -> 管理员分发
  -> 进行中
     -> 当前阶段完成
        -> 写任务日志
        -> 执行 -next -auto -type / -done / -done-plan
        -> 用 -talk 回报管理员
        -> 进入下一个任务或完成态
     -> 当前阶段阻塞
        -> 写阻塞记录
        -> 用 -talk 询问管理员或架构师
        -> 保持当前任务，等待澄清后继续
```

- 任务状态只能通过脚本流转，不能手改 `TODO.md` 或口头约定替代。
- 若只是澄清问题，不从“进行中”直接跳到“已完成”；先补记录，再等待明确回复。

## 管理员推进口径
- 优先处理 `busy` 角色回报的“已完成并已创建后续任务”消息。
- 分发前确认：任务仍在 `任务列表`、依赖已满足、角色空闲、并行人数未超上限。
- 若没有可用角色、依赖未满足或并行已满，保留任务在 `任务列表`，并用 `-talk` 反馈当前状态。
- 若计划书终验不通过，管理员只推进架构师新建的修复任务，不自行改写计划书结论。
- 若收到重复修复任务，只推进架构师确认保留的唯一任务，其余重复任务不再继续分发。
- 若执行人回报的问题实质上只是“实现未满足既有 expectation 合同”，管理员不应将原任务改为暂停；应先要求执行人继续按当前任务收口，只有出现合同层问题时才升级给架构师。
