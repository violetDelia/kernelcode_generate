# 协作执行通用规则

## 文件说明
- 功能说明：统一多角色协作时的执行顺序、任务续接、回报方式与管理员推进原则，减少角色提示词中的重复描述。
- 使用示例：角色提示词可直接引用本文件，例如：`通用执行规则以 [协作执行通用规则](../../../standard/协作执行通用规则.md) 为准。`
- 使用示例：执行人完成任务后，可先按本文件确认“记录 -> `-next -auto -type` -> `-talk` 通知管理员”的顺序是否正确。

## 通用执行规则
- 所有角色先遵守仓库根目录 `AGENTS.md`，再遵守本文件与各自提示词。
- 任务状态只能通过脚本变更，不得手工修改 `TODO.md`、`DONE.md` 或任务表内容。
- 所有角色在推进链路前，都必须先补齐任务日志或评审记录，再执行脚本或发消息。
- 若任务有独立 `worktree`，常规任务日志、阻塞记录与待确认记录必须写入该 `worktree` 下的对应记录文件；禁止在主仓根目录更新常规任务日志。
- 只有无独立任务 `worktree` 的计划互评、专题 `spec` 互评、终验或归档结论，才按规则写入计划书、专题 `spec` 正文或 `done_plan` 记录文件。
- 咨询问题统一使用 `-talk`；若只是澄清、确认、互评意见同步，不新建任务。
- 流程、权限、文件范围、任务脚本问题问管理员；实现边界、接口语义、验收口径、架构争议问架构师。
- 管理员或架构师分发任务时，`任务目标` 必须是执行人可直接落地的动作语句；至少写清“要改什么对象 / 增加什么能力 / 跑通什么测试或合同入口”，不得用“继续推进”“按当前口径收口”“整理一下”这类泛描述替代。
- 对 `build` 阶段，若 `expectation` 路径正确、入口可执行、输出明确，则默认视为当前合同真源；实现与其冲突时，执行人应优先修改实现与测试对齐，不因“实现不一致”自动回找架构师。
- 对 `build` 阶段，除 `expectation / spec / 计划书 / agents 标准文档 / immutable 内容` 等显式禁止面外，执行人默认可修改与任务目标直接相关的实现、测试和必要辅助文件；不再要求计划书为实现侧逐文件开白名单。
- `build` 只能实现当前 `spec` 已明确定义的 `API`；允许在当前文件内新增为落成该 `API` 服务的辅助函数，但不得新增未在 `spec` 明确写出的公开接口，也不得以直接调用、包装转发、别名导入、反射或其他方式使用当前文件之外的非公开 `API` 函数、方法或隐藏 helper。
- `build` 修改功能实现文件时，必须遵守 [`实现文件规范.md`](实现文件规范.md)：同步在对应文件的文件级说明中补齐或更新 `API 列表`；该列表必须紧跟在 `功能说明` 后，只做快速索引，列出当前文件承载的公开 `API` 与参数签名。若文件承载 `class`，必须列出类公开 `API`。
- `build` 修改功能实现文件时，新增或修改的函数、方法都必须按 [`实现文件规范.md`](实现文件规范.md) 补齐中文注释，至少包含 `创建者 / 最后一次更改 / 功能说明 / 使用示例`。
- 若执行人判断“要完成本轮实现必须依赖当前文件之外的非公开 `API`，或必须新增新的公开接口”才能继续，必须先暂停并回报管理员与架构师，请求调整 `spec` 或方案；未确认前不得自行推进。
- `spec` 与 `review` 阶段必须主动检查：实现中是否混入未在当前 `spec` 明确定义的非 `API` 函数、是否跨文件使用非公开 `API` 函数或方法，以及测试是否直接调用非 `API` 接口；若存在，结论不得通过。
- `review` 与复审不得以“只是内部 helper”“测试方便”“当前能跑”为由放行跨文件非公开 `API` 使用。
- 只做计划书时，架构师在形成可下发版本前，必须至少与 `3` 个对象讨论并留痕；未满足前不得建任务或通知管理员推进。
- 计划书阶段若存在争议、冲突、口径不一致或不确定事项，必须写成待确认项并回到用户确认；未确认前不得自行补假设或拍板。

## 执行前阅读规则
- 执行人不得只看 `TODO.md` 的 `info` 字段或管理员摘要就开工。
- 开工前先读：`TODO.md` 本任务行、计划书对应 `Sx` 完整章节、全局完成态/验收设计、相关 `spec / test / 功能实现` 文件；审查、修复任务还要读前序记录。
- 任务日志用一段 `执行前阅读记录` 写清已读内容、允许修改范围、禁止修改面、最低必跑命令和合同验收入口；不清楚时先记录待确认项并询问。

## 最小功能闭环规则
- 每个 `build`、重构执行或终验修复任务，至少完成一个可独立验证的功能闭环：明确行为或内部稳定接口、对应实现、对应 pytest 或可运行脚本、必要失败边界或回归用例、Diff 反推自测记录。
- 迁移类任务的最小闭环至少包括：新入口可用、旧入口残留扫描、相关导入/注册/脚本路径同步、迁移后入口测试。
- 不合格的独立执行任务包括：只改一个孤立 import、只让当前报错消失但未覆盖同类入口、只跑计划书命令而未覆盖实际 diff、只写日志或只写 TODO。
- `spec`、计划互评、审查和合并任务不要求实现闭环，但必须完成各自闭环：规格可指导执行、审查可定位风险、合并可追溯记录与基线。

## 自检规则
- 除 `merge` 角色外，所有 `spec / build / review / 替补 / 终验修复` 执行人在推进前必须完成自检，并把自检结果写入任务记录。
- 通用自检至少回答：是否已读完整阶段说明、是否未越权改文件、是否完成本阶段闭环、是否按实际 diff 做了对应测试或审查、是否解决了原问题、是否满足全部要求、是否仍有已知 bug / 逻辑问题 / 未覆盖边界 / 潜在漏洞。
- `spec` 自检重点：接口是否合理，是否存在功能冗余、重复定义或维护成本过高的问题；功能边界、异常情况、限制条件、兼容性 / 破坏性变更、弃用口径是否写全；性能、可靠性、安全、资源生命周期等非功能影响是否说明；文字是否一眼能看懂，是否存在歧义。
- `spec` 自检必须额外确认：当前 `spec` 是否在“功能简介”后紧跟 `API 列表`；`API 列表` 是否只做快速索引且已列出本轮允许实现的 `API` 集合；每个 `API` 是否写清参数签名；若是 `class` 场景，是否逐条列出了类公开 `API`；是否没有给实现侧留下“可自行新增函数/隐藏接口”的口子；若已有实现或测试，是否已经检查并标出非 `API` 函数与测试直连非 `API` 接口的问题。
- `build` 自检重点：实现是否有遗漏；代码是否冗余；注释、示例和函数说明是否准确且不过时；是否有可复用逻辑未抽取；是否存在过大的函数、不合理的一两行小函数、职责混杂或影响后续维护的结构问题；输入/输出校验、并发、资源释放、错误处理、性能、兼容性是否被考虑。
- `build` 自检必须额外确认：本轮若新增 helper，仅限当前文件内辅助函数；没有新增未在当前 `spec` 明确定义的公开接口；没有以直接调用、包装转发、别名导入、反射或其他方式使用当前文件之外的非公开 `API` 函数、方法或隐藏 helper；测试也没有转而直连非 `API` 接口来绕过公开行为。
- `build` 自检必须额外确认：对应功能实现文件的文件级说明里是否已补齐或更新 `API 列表`；该列表是否紧跟在 `功能说明` 后、是否只保留当前文件公开 `API` 与参数签名；若是 `class` 文件，类公开 `API` 是否逐条列出。
- `build` 自检必须额外确认：新增或修改的每个函数、方法是否都已按 [`实现文件规范.md`](实现文件规范.md) 补齐中文注释；注释中的功能说明与使用示例是否与当前实现一致。
- `build` 的测试自检必须确认测试有意义：断言能在实现坏掉时失败，覆盖主路径、边界、异常和回归点；测试本身不能复杂到难维护。
- `review` 自检重点：主路径、特殊情况、失败边界和内容完整性是否都看过；冗余、维护性、扩展性、可读性、代码结构和测试证据是否足够；是否逐行理解了被审的人写代码，并结合整文件和系统上下文判断；凡是不完美、可修改、可降低风险的点都必须写出来，不得因为“能跑”就放过。
- `review` 自检必须额外确认：是否逐项检查了新增/修改函数是否都属于当前 `spec` 明确列出的 `API`；是否检查了实现未以任何方式使用当前文件之外的非公开 `API`，测试未直接调用非 `API` 接口；若发现任一项，必须写成 `需修改`。
- `review` 遇到安全、并发、性能、隐私、架构或领域知识不足时，必须写明并要求领域专家补审；任务过大、混入多个功能或复杂度过高时，应要求拆分。
- 计划书自检重点：示例是否清晰易懂，执行人只读阶段正文能否开工；公开 `API` 与参数顺序是否合理；方案是否是当前最优取舍；任务边界、依赖、验收资产和非目标是否明确；是否写清兼容性、风险、回滚或失败例外；任何歧义、冗余或可改进点都要在互评前处理或列为待确认。
- 计划书自检必须额外确认：是否已经完成不少于 `3` 个对象的讨论记录；凡是争议、冲突、口径不一致或不确定事项，是否都已写成待确认项并回到用户确认；若尚未得到用户确认，是否明确保持未决状态而非擅自定案。
- 自检不得只写“已自检”“无问题”或复制规则标题；必须写出实际检查结论。若存在未覆盖项，必须写清原因、风险和下一步处置。
- 缺自检记录时，审查人、管理员或合并角色应要求补齐后再推进。

## Diff 反推测试规则
- `build`、重构执行、审查与复审不能只执行计划书、任务树或任务描述中列出的验收命令；这些命令只是最低集合。
- 执行人与审查人都必须基于本轮实际改动文件反推测试；可使用 `git diff --name-only`、任务 worktree 中的实际文件清单，或审查对象给出的等价 diff 清单。
- `Diff 反推测试` 只指 `pytest`、项目测试脚本、工具测试脚本或可作为测试运行的本地脚本；`expectation` 是合同验收资产，不计入 `Diff 反推测试`，也不能替代改动文件对应测试。
- 每个实际改动文件都要找到对应测试：优先使用文件注释、`spec/test/功能实现` 链接或相邻测试；其次按 [`测试文件约定`](测试文件约定.md) 中“改动文件到测试入口默认映射”；最后按目录最近的 `pytest` 或测试脚本入口。
- 如果改动涉及公开输出、IR 文本、helper 名称、op 名称、导入路径、目录迁移、脚本路径或错误文本，除反推测试外，可按合同真源要求单列补跑对应 `expectation`，并执行必要的 `rg` 残留扫描；这类合同验收不得写成已覆盖反推测试。
- 执行阶段必须在任务日志 `验证` 字段中写出 `Diff 反推自测`，至少包含：实际改动文件、反推出的测试命令、执行结果、未执行项与原因。
- 审查阶段必须在审查日志或审查记录中写出 `Diff 反推审查`，至少包含：被审 diff 文件、反推出的测试命令、审查复跑或核验证据、缺失项判断。
- 若 diff 反推出的测试失败，默认不得继续标记完成或审查通过；只有失败已被证明为本轮前已存在、与本轮改动无关，且记录了基线证据和架构师明确认可，才可作为已知缺口继续流转。
- 若执行人未提供 `Diff 反推自测`，审查人应判定为 `需修改`；若审查人未提供 `Diff 反推审查`，管理员或合并角色应要求补齐后再推进。

## 任务流转规则
- 计划书、专题 `spec` 与小任务由架构师互评通过后建任务；计划书未互评通过前不得建任务或通知管理员推进。
- 计划书在进入互评完成态前，必须同时满足：`>=3` 个讨论对象记录已齐全，且所有争议 / 冲突 / 不确定事项都已得到用户确认或保持为明确未决项；否则不得建任务或通知管理员推进。
- `spec / build / review / 替补` 完成当前任务后，顺序固定为：写记录 -> `-next -auto -type` -> `-talk` 通知管理员。
- `-next -auto -type` 只用于当前任务已完成且需要下游任务的场景；任务仍阻塞、待确认或未完成时，只写阻塞记录并 `-talk`。
- `merge` 由 `李白` 独立完成：写合并日志 -> 合并 -> 当前任务 `-done` -> `-talk` 回报。
- 管理员分发前只核对三件事：任务在列表、依赖满足、角色空闲且未超并行上限。

## 终验与归档规则
- 计划书终验、复验或终验修复复核必须基于最新同步现场；若主目录不能对齐最新主线，必须写明替代现场、`commit` 与执行目录。
- 架构师终验必须运行与本轮改动有关的 `expectation` 合同验收，并把结论写回计划书正文，至少包含结论人、验证基线、执行目录、通过摘要或最小阻断项；若用户另有明确要求全量 `expectation`，则按用户要求执行。
- 计划书终验、复验或终验修复复核的通过与否，只基于计划正文当前保留的预期功能、公开 `API` 与当前验收资产；`done_plan`、历史记录、对齐记录、归档说明、旧阶段编号与文字同步问题，若不影响这三项，不得单独作为“不通过”理由。
- 终验不通过不得归档；架构师围绕最小阻断项补建修复任务，管理员只推进该修复任务。
- 修复任务完成后回到原计划书重新验收；双架构师均通过后，管理员先创建归档任务，再执行 `-done-plan`。
- 若出现重复修复任务，只保留一个范围清楚、依赖正确、`worktree` 明确的任务继续推进。
