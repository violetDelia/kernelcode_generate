# 合并规范

## 文件说明
- 功能说明：统一 `merge` 阶段允许带入的文件范围、禁止项与最小执行顺序，避免合并任务顺手带入越界内容。
- 使用示例：`李白` 合并前，可先看本文件中的“允许合并文件类型白名单”，确认当前提交是否只包含允许项。
- 使用示例：审查人发现提交里混入 `TODO.md` 或 `skills/` 文件时，可直接按本文件判定越界。

## 通用规则
- 禁止在合并流程中使用 `git stash`。
- 合并相关网络命令统一使用 `timeout 60` 窗口（含 `git push`；如需执行 `git fetch` 亦遵守该窗口）；超时按失败处理，且不重试。
- `李白` 只在任务指定的 `worktree` 内执行合并；不得回到仓库主目录做本地合并。
- 合并完成态为：在指定 `worktree` 内完成合并提交，并向远端主分支发起一次提交。
- 合并默认不执行测试；仅在出现冲突时按最小范围执行必要测试。
- `git push` 失败（如网络超时）视为任务成功；失败后停止重试，并在任务回报中说明阻塞与待补推提交。
- 合并前必须确认目标 `worktree` 内不存在其他进行中任务；若仍有未完成任务，禁止合并。
- `agents/` 目录内除 `task_records/` 下的日志文件外，其他文件仅在当前 `merge` 任务明确点名且已通过对应审查时才允许带入；未点名的共享状态文件一律禁止。
- 合并内容不得包含 `TODO.md`、`DONE.md`、`AGENTS.md`，以及 `skills/` 目录中的文件或子目录。
- 冲突处理以最新有效改动为准。
- 合并提交信息必须使用格式 `T-<task_id>-<desc>`。
- `expectation/` 路径属于合同资产，默认不纳入常规合并；仅当当前任务明确属于 `expectation` 合同收口，且改动由架构师或被架构师明确授权的规格阶段产出时，才允许按任务边界带入，具体以 [`expectation任务规则.md`](expectation任务规则.md) 为准。
- 合并后若改工作树无其他任务，则删除该工作树。无需再追加工作日志。
- 合并成功不再追加记录文件日志；仅在合并失败或阻塞时追加说明。
- 合并流程不再要求合并前 `git fetch`；如需更新远端信息，仅在合并完成后再执行 `git fetch`。`git fetch` 超时只记录，不阻断合并流程。
- 合并任务以合并提交完成为收口点；完成后可尝试一次 `git fetch`，失败不影响任务完成判定。

## 允许合并文件类型白名单

| 文件类型 | 是否允许 | 说明 |
| --- | --- | --- |
| 当前任务点名的实现文件 | 允许 | 仅限当前 `merge` 任务对应链路已审查通过的实现改动 |
| 当前任务点名的测试文件 | 允许 | 仅限与本链路直接相关的测试 |
| 当前任务点名的标准文档或计划文档 | 允许 | 仅当任务范围明确点名这些文档 |
| `agents/codex-multi-agents/log/task_records/...` | 允许 | 仅允许带入当前链路对应的任务记录文件 |
| `TODO.md` | 禁止 | 任务状态文件，不属于合并交付物 |
| `DONE.md` | 禁止 | 完成态记录文件，不属于合并交付物 |
| `AGENTS.md` | 禁止 | 仓库根规则文件，不应被常规合并任务顺带修改 |
| `skills/` 目录 | 禁止 | 本轮治理范围外，且标准规则已明确不得顺手修改 |
| `agents/` 下非 `task_records` 的其他共享状态文件 | 禁止 | 如 `agents-lists.md`、`talk.log`，不应随业务合并进入 |
| `expectation/` 路径 | 条件允许 | 仅当当前任务明确属于 `expectation` 合同收口，且改动由架构师或被架构师明确授权的规格阶段产出；其余情况一律禁止，具体以 [`expectation任务规则.md`](expectation任务规则.md) 为准 |

## 白名单使用方式

- 先按任务描述确认本轮允许的业务范围，再用白名单判断提交里的每一类文件是否可带入。
- 若发现 `task_records` 之外的 `agents/` 文件混入合并，必须先确认该文件是否被当前任务明确点名；未点名则视为越界。
- 若发现提交中包含 `expectation/` 文件，必须先对照 [`expectation任务规则.md`](expectation任务规则.md) 核对任务范围、角色与授权来源；任一条件不满足即不得带入。
- 白名单只覆盖当前标准文档已定义的常规可交付类型；不在表内的文件，默认按“不允许”处理，先回报管理员或架构师。
