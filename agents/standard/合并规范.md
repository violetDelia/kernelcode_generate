# 合并规范

## 文件说明
- 功能说明：统一合并前核对、允许带入文件、敏感文件权限和合并记录。
- 使用示例：`merge` 任务开始前，按本文件逐项确认。

## 合并前核对
- 非计划级任务已通过审查；计划级任务已通过架构复核 / 终验。
- 任务记录完整，且包含验证摘要、自检或审查结论。
- 合并记录必须在合并提交前写入当前链路任务记录，并与代码 / `spec` / 测试等任务改动同批合入；不得先合代码再补一个记录提交。
- 审查、复审、架构复核和终验提出的所有可执行返工项必须有收口证据；未收口时退回管理员接续，不得合并。
- 合入范围只包含当前任务链允许文件。
- 公开 `API` 变更有用户确认来源。
- `expectation/` 改动只能来自用户或架构师确认来源 / 明确授权链的合同任务，并同时有架构师记录的授权范围、manifest / hash 或 scope 外空 diff 核对、合同目标和合同验收结果。
- 若存在未通过测试或合同验收，记录中必须写清基线证据和当前认可口径。

## 允许带入
| 文件类型 | 是否允许 | 条件 |
| --- | --- | --- |
| 当前任务相关实现 | 允许 | 已审查通过 |
| 当前任务相关测试 | 允许 | 已审查通过 |
| 当前任务相关 `spec` | 允许 | 与公开 API 确认来源一致 |
| 当前任务记录 | 允许 | 当前链路对应记录文件 |
| 计划书 / 标准文档 / 提示词 | 条件允许 | 当前任务明确点名 |
| `expectation/` | 仅授权合同任务允许 | 非架构角色不得带入普通任务改动；必须有用户或架构师确认来源 / 明确授权链、架构师记录的授权范围、合同目标、范围核对和验收记录 |
| `TODO.md` / `DONE.md` | 禁止手工带入 | 状态只能通过脚本处理 |

## 禁止项
- 不合入未审查通过的顺手修复。
- 不合入未点名的角色提示词、标准文档或共享状态文件。
- 不合入无授权来源的 `expectation/` 改动。
- 不用合并任务补做实现、测试或架构裁定。
- 不覆盖主仓或其他任务的本地改动。

## 合并记录
- 合并记录必须写清实际合入来源、源提交或源工作树、冲突处理、验证命令和结果；最终合并提交号可在合并后回报中说明，不要求为补最终提交号再追加记录提交。
- 合并记录和代码必须在同一个合并 diff / commit 中出现；合并后只执行状态脚本和回报，不再为补任务记录追加单独提交。
- 若合并前发现记录缺失、记录不在待合入 diff 内或记录与代码不对应，先暂停补齐，不得先合代码。
- 若未运行验证，必须写明原因和风险。
- 若主仓同步受本地状态阻塞，只记录同步阻塞，不覆盖现有内容。
- 合并失败时写清失败命令、退出码、冲突文件和下一步建议。
