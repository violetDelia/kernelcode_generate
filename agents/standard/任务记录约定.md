# 任务记录约定

## 文件说明
- 功能说明：统一任务记录字段、落点、验证摘要和自检写法。
- 使用示例：推进、回退、阻塞、终验或合并前，先补齐本文件要求的记录。

## 通用规则
- 记录先于状态推进、脚本执行和回报消息。
- 常规任务记录写入对应任务 `worktree`；无独立 `worktree` 的架构结论写入计划书、专题 `spec` 或 `done_plan`。
- 任务状态文件路径以管理员下发为准；脚本示例中的 `TODO.md` 只是占位，独立 `worktree` 内不存在时使用主仓绝对路径或先问管理员。
- 同一任务链沿用同一个记录文件，按实际时间追加。
- 阻塞、待确认、修复接续、审查结论、终验结论和合并结果都必须写记录。
- 多人讨论、返工意见和用户 / 架构师确认也必须写入正式记录或计划正文。
- 不得只在会话、`talk.log` 或口头回报中保留关键结论。

## 记录落点生命周期
- 任务执行、审查、阻塞和返修期间，记录默认写入任务 `worktree` 内的指定记录文件。
- 合并或归档后，主仓 `agents/codex-multi-agents/log/task_records/...` 与 `done_plan` 中的落位记录是存活记录。
- 临时 `worktree` 被回收后，不得再把缺少旧 `worktree` 路径当作阻断；必须先核对主仓存活记录是否已经落位。
- 若主仓存活记录缺失或与任务链记录不一致，才按记录缺失处理并暂停推进。
- 存量旧任务接续时，最少补齐状态文件路径、当前阶段、责任角色、存活记录路径、最新同步现场、未收口返工、`expectation` scope，以及是否需要补五行短口径。

## 强制字段与顺序
- `时间`：实际记录时间，推荐精确到分钟。
- `经办人`：当前角色名。
- `任务`：任务号、计划名或阶段名。
- `任务目标`：本轮直接产出，不写长期规划。
- `改动`：实际修改、审查结论、裁定或阻塞定位。
- `验证`：命令、脚本、核对方式、结果和退出码；未执行写原因。
- `自检`：非 `merge` 任务必填，写本轮实际结论。
- `结论`：通过、需修改、阻塞、待确认、已合并或下一步接手方。

## 执行记录要求
- `执行前阅读记录` 写已读任务、计划书、计划内小任务卡、验收设计、相关 `spec/test/实现` 与前序记录。
- `最小功能闭环` 写本轮交付行为、实现入口、测试入口、失败边界和未覆盖原因。
- `Diff 反推自测` 写实际改动文件、反推测试命令、执行结果、未执行项和风险。
- `自检` 写接口、边界、异常、兼容、实现遗漏、冗余、注释、复用、函数粒度、输入输出、资源、并发、性能和测试有效性。
- `返工收口` 写审查或终验提出的问题、实际修复动作、对应验证和仍未覆盖项。

## 审查记录要求
- `review`、复审和终验修复复核必须写最新同步现场：执行目录、目标分支或基线、`fetch` / 对齐或 `merge-base` 核对结果、是否存在冲突或覆盖风险。
- `Diff 反推审查` 写被审 diff 文件、反推测试命令、复跑或核验证据、未覆盖项。
- findings 按严重程度写文件位置、影响、最小修复建议和验收方式。
- 必须核对执行人的执行前阅读、最小功能闭环、自检和 diff 反推自测。
- 缺关键记录或测试与 diff 不匹配时，结论不得为 `通过`。
- 可读性、可维护性、测试有效性和边界完整性问题必须写成返工项，不得只写“建议优化”。
- 第二轮及以后返工必须标注 `新增问题 / 重复问题 / 范围扩大`，并写清是否需要架构裁定或用户确认。
- 同一阻断项重复出现，或进入第三轮返工前，记录中必须写清管理员已转架构师裁定根因，或说明仍缺哪项裁定。

## 计划与终验记录要求
- 计划互评写评审人、讨论问题、证据、分歧、采纳结论、最小需改项和是否可下发。
- 终验、复验或修复复核写结论人、验证基线、执行目录、当前必过合同验收命令、结果和最小阻断项。
- full expectation 或 runner 类验收必须额外写 `cwd`、`PYTHONPATH`、`EXPECTATION_WORKTREE_ROOT`、sync worktree、授权 `expectation` scope、runner 特例、timeout 和日志路径。
- 历史 / 本地只读 `expectation` 来源只写说明，不写成当前必过验收命令。
- 结论必须回写计划书或专题 `spec` 正文。

## 合并记录要求
- 合并记录必须在合并提交前写入当前链路任务记录，记录实际合入文件、来源提交或工作树、验证、冲突处理、敏感文件核对和剩余风险。
- 合并记录必须与代码、`spec`、测试等任务改动同批合入；不得先提交或推送代码，再补一个任务记录提交。
- 最终合并提交号可在合并后回报中说明，不要求为补最终提交号再追加记录提交。
- 若合并阶段发现记录缺失或记录未包含在待合入 diff 中，先补齐并重新核对，再执行合并 / push / `-done`。
- 合并完成后的状态脚本和回报只记录状态结果，不替代同批合入的合并记录。

## 验证写法
- `expectation` 单列为 `合同验收`，不计入 diff 反推测试。
- 失败项必须写复现命令、失败摘要、是否与本轮改动相关和当前判断。
- 未运行命令必须写清原因，不能写“未执行”后不说明。
- 验证必须说明断言或核对点锁定了什么行为；仅写命令名和退出码不足以证明测试有效。

## 最小模板
```md
时间：YYYY-MM-DD HH:MM
经办人：<name>
任务：<task>
任务目标：<goal>
改动：<files / decision / finding>
验证：<commands / result / exit code>
自检：<non-merge only>
结论：<pass / need change / blocked / next>
```
