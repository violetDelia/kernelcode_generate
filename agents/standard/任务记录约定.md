# 任务记录约定

## 文件说明
- 功能说明：统一任务链记录的最小字段、更新时间点与写法，保证管理员、架构师、审查人都能直接接手。
- 使用示例：执行人完成当前阶段后，先按本文件补全记录，再执行 `-next -auto -type ...`。
- 使用示例：`李白` 开始合并前，先按本文件写完工作日志，再执行合并与回报。

## 通用规则
- 所有任务记录都必须写入对应记录文件；计划互评、专题 `spec` 互评、最终验收也属于任务记录的一部分。
- 若任务有独立 `worktree`，常规任务日志必须创建或追加在该 `worktree` 下的对应记录文件；同一任务链后续记录也必须继续写回该 `worktree` 中的同一记录文件。
- 禁止在主仓根目录更新常规任务日志；只有无独立任务 `worktree` 的计划互评、专题 `spec` 互评、终验或归档结论，才按规则写入计划书、专题 `spec` 正文或 `done_plan` 记录文件。
- 若本轮属于计划书终验、复验或终验修复复核，给出结论的架构师必须把检验报告或终验结论写回计划书正文；不得只把结论停留在会话、`talk.log` 或单独口头回报里。
- 同一任务链默认沿用同一个记录文件，内容按实际执行顺序追加，不得打乱先后关系。
- 任何会改变任务状态或触发链路推进的动作，都必须先写记录，再执行脚本或发回报消息。典型动作包括：`-next`、`-done`、通知管理员推进、通知管理员归档。
- 任务阻塞时，也必须先写一条阻塞记录，再回报管理员或架构师。
- `李白` 的工作日志必须在开始合并前写完；若合并阻塞，先补阻塞记录，再回报管理员。
- 计划归档必须先有独立归档任务；归档记录统一写入 `agents/codex-multi-agents/log/task_records/done_plan/<年份>/<周>/` 下对应记录文件。

## 强制字段与顺序
- 每条记录必须按以下顺序书写：`时间`、`经办人`、`任务`、`任务目标`、`改动`、`验证`、`结论`；非 `merge` 任务在 `结论` 前补 `自检`。
- `验证` 为强制字段；若本轮没有执行命令，应明确写 `未执行命令，原因：...`。
- 除 `merge` 任务外，执行记录必须包含 `执行前阅读记录` 与 `自检`；`execute`、重构执行和终验修复还必须包含 `最小功能闭环`。
- `执行前阅读记录` 写已读任务行、计划书正文、全部计划内小任务卡、全局验收设计、相关 `spec/test/实现` 与前序记录；计划级 `execute` 任务必须说明本轮覆盖了哪些 `Sx` 小任务卡。
- `最小功能闭环` 写本轮行为、实现入口、测试入口、失败边界或未覆盖原因。
- `自检` 先写通用结论：已读完整阶段、未越权改文件、闭环已完成、测试/审查已按 diff 反推、原问题已解决、要求已满足、无已知 bug / 逻辑问题 / 未覆盖边界 / 潜在漏洞；再按角色补充 `execute / review / 计划书` 的重点自检结论。
- `execute`、重构执行、审查与复审的 `验证` 字段必须包含按实际 diff 反推的记录：执行阶段写 `Diff 反推自测`，审查阶段写 `Diff 反推审查`。
- `Diff 反推自测 / 审查` 不得把 `expectation` 算作测试；另跑 `expectation` 时单列为“合同验收”。
- 若需要帮助下游接手，可在 `结论` 后追加 `后续建议`，但不得省略前面的强制字段。

## 字段写法
- `时间`：写实际记录时间，推荐精确到分钟。
- `经办人`：写当前执行角色名。
- `任务`：写任务号或当前阶段名。
- `任务目标`：只写本轮直接产出，不写长期规划。
- `改动`：写本轮实际修改的文件、判断、审查结论或阻塞定位；不写空泛描述。
- `验证`：写命令、脚本、检查方式和结果；若只做文本核对，也要写清核对对象。
- `结论`：写通过/不通过/阻塞/待确认，以及下一步该由谁接手。

## 自检写法
- 禁止只写“已自检”“无问题”或照抄规则标题；必须写本轮实际结论。
- `execute` 示例：`自检：API 参数顺序与返回语义已核对；计划内 S1/S2 已按执行步骤完成；实现覆盖主路径、非法输入和回归 case；函数注释与示例已同步；未新增过大函数或无意义小函数；资源释放和错误处理已检查；测试断言会在实现坏掉时失败。`
- `review` 示例：`自检：已逐行审查改动文件并结合调用链核对；特殊情况、异常、兼容、测试有效性和可维护性已检查；发现 2 个可改进点并列为最小需改项；未涉及安全/并发专家补审。`
- `计划书` 示例：`自检：示例、API、边界、依赖、非目标和验收资产已机械可判；计划内小任务拆分不过大；兼容风险和失败例外已写清；执行人只读对应小任务卡可开工。`


## 最小合格记录

```
时间：2026-04-15 10:20
经办人：金铲铲大作战
任务：T-20260415-xxxx
任务目标：补齐标准规则组执行前自检规则
执行前阅读记录：已读 TODO 本计划级任务行、计划书正文、当前计划内小任务卡、全局验收设计与 agents/standard 相关文件
最小功能闭环：标准规则新增执行前自检要求，任务记录模板和角色提示词都能对应落地
改动：更新 agents/standard/协作执行通用规则.md，新增“自检规则”与执行记录要求
验证：rg -n "自检规则|执行前阅读记录" agents/standard agents/codex-multi-agents/agents -> 命中新增规则与角色提示词
Diff 反推自测：改动文件为 agents/standard/协作执行通用规则.md；反推验证为 git diff --check 与 rg 文本核对；结果通过
自检：已读完整阶段；未越权改文件；最小闭环已完成；已按 diff 反推做文本校验；本轮为标准文档改动，已检查文字无歧义、角色可执行、无旧状态图残留
结论：当前 execute 已完成，任务日志已写入对应任务 worktree 的记录文件；下一步创建 review 任务并通知管理员推进
```

## 阻塞记录

```
时间：2026-04-15 10:45
经办人：金铲铲大作战
任务：T-20260415-yyyy
任务目标：补齐标准规则组异常分类
改动：已核对计划书要求与当前 worktree，发现目标文件缺失，无法继续补正文；已确认不属于当前角色可自行扩范围创建的文件
验证：test -f agents/standard/异常处理规范.md -> 未通过；rg -n "异常分类" ARCHITECTURE/plan/demo.md -> 命中当前阶段要求
结论：当前任务阻塞，已写阻塞记录；下一步用 -talk 向管理员或架构师确认缺失文件与允许修改范围
```

## 合并任务补充
- 合并任务的 `改动` 应写清本次实际合入的 `worktree`、分支或冲突处理结果。
- 合并任务的 `验证` 应写清是否执行了最小必要验证；若未执行，应写明原因。
- 合并任务的 `结论` 应明确写“已合并，已执行 `-done`，请管理员继续推进”或“合并阻塞，请管理员处理后续”。
