# 任务记录约定

## 文件说明
- 功能说明：统一任务链记录的最小字段、更新时间点与写法，保证管理员、架构师、审查人都能直接接手。
- 使用示例：执行人完成当前阶段后，先按本文件补全记录，再执行 `-next -auto -type ...`。
- 使用示例：`李白` 开始合并前，先按本文件写完工作日志，再执行合并与回报。

## 通用规则
- 所有任务记录都必须写入对应记录文件；计划互评、专题 `spec` 互评、最终验收也属于任务记录的一部分。
- 同一任务链默认沿用同一个记录文件，内容按实际执行顺序追加，不得打乱先后关系。
- 任何会改变任务状态或触发链路推进的动作，都必须先写记录，再执行脚本或发回报消息。典型动作包括：`-next`、`-done`、通知管理员推进、通知管理员归档。
- 任务阻塞时，也必须先写一条阻塞记录，再回报管理员或架构师。
- `李白` 的工作日志必须在开始合并前写完；若合并阻塞，先补阻塞记录，再回报管理员。
- 计划归档必须先有独立归档任务；归档记录统一写入 `agents/codex-multi-agents/log/task_records/done_plan/<年份>/<周>/` 下对应记录文件。

## 强制字段与顺序
- 每条记录必须按以下顺序书写：`时间`、`经办人`、`任务`、`任务目标`、`改动`、`验证`、`结论`。
- `验证` 为强制字段；若本轮没有执行命令，应明确写 `未执行命令，原因：...`。
- 若需要帮助下游接手，可在 `结论` 后追加 `后续建议`，但不得省略前面的强制字段。

## 字段写法
- `时间`：写实际记录时间，推荐精确到分钟。
- `经办人`：写当前执行角色名。
- `任务`：写任务号或当前阶段名。
- `任务目标`：只写本轮直接产出，不写长期规划。
- `改动`：写本轮实际修改的文件、判断、审查结论或阻塞定位；不写空泛描述。
- `验证`：写命令、脚本、检查方式和结果；若只做文本核对，也要写清核对对象。
- `结论`：写通过/不通过/阻塞/待确认，以及下一步该由谁接手。

## 记录模板

```
时间：
经办人：
任务：
任务目标：
改动：
验证：
结论：
```

## 合并任务补充
- 合并任务的 `改动` 应写清本次实际合入的 `worktree`、分支或冲突处理结果。
- 合并任务的 `验证` 应写清是否执行了最小必要验证；若未执行，应写明原因。
- 合并任务的 `结论` 应明确写“已合并，请管理员执行 `-done`”或“合并阻塞，请管理员处理后续”。

## 简短示例

```
时间：2026-04-11 10:20
经办人：小李飞刀
任务：T-20260411-xxxx
任务目标：补齐 execute_engine 调用路径实现与测试
改动：更新 kernel_gen/execute_engine/execution_engine.py 与 test/execute_engine/test_execute_engine_invoke.py
验证：pytest -q test/execute_engine/test_execute_engine_invoke.py -> 12 passed
结论：当前 build 已完成，任务日志已写完；下一步创建 review 任务并通知管理员推进
```

```
时间：2026-04-11 14:05
经办人：李白
任务：T-20260411-yyyy
任务目标：合并 execute_engine S2 已通过审查的改动
改动：从 wt-20260411-exec-engine-s2 合并指定文件；无额外改动
验证：未执行命令，原因：当前任务无额外验证要求，沿用审查记录
结论：已合并，请管理员执行 -done
```
