# 实现文件规范

## 文件说明
- 功能说明：统一功能实现文件与函数注释格式，明确文件级 `API 列表`、函数注释模板与跨文件 `API` 边界，减少实现、审查与互评时的口径分歧。
- 使用示例：修改 `kernel_gen/**`、`include/**`、`agents/**` 中的功能实现文件前，先按本文件补齐文件级说明与函数注释。
- 使用示例：审查实现时，先按本文件核对“文件级 `API 列表` 是否齐全”“函数注释是否完整”“跨文件调用是否只使用 `spec` 定义的公开 `API`”。

## 适用范围
- 本文件适用于仓库中的功能实现文件，包括但不限于：
  - `kernel_gen/**`
  - `include/**`
  - `agents/**` 下承载实际功能实现的脚本或模块
- `test/**` 仅在函数注释风格上参考本文件；测试额外约束以 [`测试文件约定.md`](测试文件约定.md) 为准。
- `spec/**` 不适用本文件结构，`spec` 写法以 [`spec文件规范.md`](spec文件规范.md) 为准。

## 文件级说明要求
- 每个功能实现文件都必须在文件头提供完整说明。
- 文件级说明必须使用**中文**，并至少包含以下章节，顺序固定：
  - `创建者`
  - `最后一次更改`
  - `功能说明`
  - `API 列表`
  - `使用示例`
  - `关联文件`
- `API 列表` 必须紧跟在 `功能说明` 后，不得后移。
- `API 列表` 只列当前文件承载的公开 `API`，不列当前文件私有 helper，也不列当前文件之外的接口。
- 新增、删除、重命名或修改公开 `API` 前，必须先取得用户明确同意；没有用户确认来源时，不得通过修改文件级 `API 列表` 把 helper、别名、兼容入口或脚本参数升级为公开 `API`。
- `API 列表` 中的每个公开 `API` 必须写完整参数签名；格式必须类似：
  - `` `foo(arg1: Type1, arg2: Type2) -> ReturnType` ``
  - `` `class Foo(arg1: Type1, arg2: Type2)` ``
  - `` `Foo.run(arg1: Type1, arg2: Type2) -> ReturnType` ``
- 若文件承载 `class`，必须列出：
  - `class` 构造签名
  - 该类的全部公开方法签名
  - 必要时可列公开属性，但不得把私有属性或内部辅助方法混入 `API 列表`
- `关联文件` 至少列出：
  - `spec`
  - `test`
  - `功能实现`
- `关联文件` 必须使用仓库内可点击路径，不得只写纯文本文件名。

## 文件级说明模板

```python
"""Compiler hook for ExecutionEngine (P0/S2).

创建者: 朽木露琪亚
最后一次更改: 小李飞刀

功能说明:
- 提供编译命令生成与编译单元拼接的最小实现，使 S2 阶段可稳定覆盖“编译路径”合同。
- 默认以 dry-run 形式生成编译命令与产物占位文件，不强制依赖真实编译器环境。

API 列表:
- `default_compiler() -> str`
- `build_compile_unit(source: str, target_includes: Iterable[str], entry_shim_source: str) -> CompileUnit`

使用示例:
- from kernel_gen.execute_engine.compiler import default_compiler, build_compile_unit
- assert default_compiler() == "g++"
- unit = build_compile_unit(source="int main(){}", target_includes=(), entry_shim_source="")

关联文件:
- spec: [`spec/execute_engine/execute_engine_target.md`](../../spec/execute_engine/execute_engine_target.md)
- test: [`test/execute_engine/test_execute_engine_compile.py`](../../test/execute_engine/test_execute_engine_compile.py)
- 功能实现: [`kernel_gen/execute_engine/compiler.py`](../../kernel_gen/execute_engine/compiler.py)
"""
```

## 函数注释要求
- 每个新增或修改的函数、方法都必须有**中文**注释。
- 函数注释至少包含以下内容，顺序固定：
  - `创建者`
  - `最后一次更改`
  - `功能说明`
  - `使用示例`
- 函数注释不强制重复写 `spec/test/功能实现` 链接；这些链接由文件级说明统一承载。
- 函数注释中的 `功能说明` 必须描述当前函数真实职责，不得写成“调用某 helper”“做一些处理”这类空泛文字。
- `使用示例` 必须能体现当前函数的实际调用方式；对纯内部 helper，可写最小可运行示例。
- 公共 `API` 函数的注释不得与 `spec` 定义冲突；若参数、返回、边界行为发生变化，必须同步更新注释和 `spec`。

## 函数注释模板

```python
def build_compile_unit(
    source: str,
    target_includes: Iterable[str],
    entry_shim_source: str,
) -> CompileUnit:
    """创建最小编译单元描述。

    创建者: 朽木露琪亚
    最后一次更改: 小李飞刀

    功能说明:
    - 组合源码、目标 include 和入口 shim 文本，生成后续编译阶段使用的 `CompileUnit`。

    使用示例:
    - unit = build_compile_unit(source="int main(){}", target_includes=(), entry_shim_source="")
    - assert unit.source == "int main(){}"
    """
```

## 跨文件 API 边界
- 当前文件之外，**只允许**调用 `spec` 已定义且已在对应文件 `API 列表` 中列出的公开 `API`。
- 未在 `spec` 中定义、未在文件级 `API 列表` 中列出的函数、方法、类、别名、helper、隐藏入口，都视为非公开 `API`。
- 非公开 `API` 不得通过以下方式跨文件使用：
  - 直接导入
  - 包装转发
  - 别名导入
  - 反射访问
  - 通过测试绕过公开入口直连内部 helper
- 若要跨文件复用某个能力，必须先在 `spec` 中把它升级为公开 `API`，再同步更新对应实现文件 `API 列表` 与测试。
- 将能力升级为公开 `API` 前，还必须先取得用户明确同意；未确认前只能保留为当前文件内 helper 或写成待确认项。

## 审查与自检重点
- `execute` 自检必须确认：
  - 本轮新增/修改函数是否都补齐了函数注释。
  - 对应实现文件是否已补齐文件级 `API 列表`，且紧跟在 `功能说明` 后。
  - `API 列表` 是否只列公开 `API`，且签名完整。
  - 本轮若新增、删除、重命名或修改公开 `API`，是否有可追溯的用户确认来源。
  - 是否存在跨文件调用非公开 `API` 的行为。
- `review` / 互评必须确认：
  - `spec` 是否真的定义了当前实现跨文件使用的 `API`。
  - 公开 `API` 变更是否有用户确认来源；没有时不得通过。
  - 测试是否只验证公开 `API` 或明确声明的稳定契约。
  - 注释、示例、`API 列表` 与当前实现是否一致。
