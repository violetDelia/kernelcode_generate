# 实现文件规范

## 文件说明
- 功能说明：统一功能实现文件、公开 `API`、注释、跨文件调用和测试边界。
- 使用示例：新增或修改 `kernel_gen/**` 等功能文件前，先按本文件核对。

## 文件级说明
- 每个功能实现文件必须有文件级说明。
- 文件级说明必须包含：`功能说明`、`API 列表`、`使用示例`、`关联文件`。
- `API 列表` 必须紧跟在 `功能说明` 后，只列当前文件承载的公开 `API` 与参数签名。
- class 场景必须逐条列类公开方法签名。
- 文件级说明应反映当前真实行为，不保留失效迁移说明或过程记录。
- 人员元信息不是文件级说明要求；不得因为缺少人员元信息退回实现文件。

## 函数注释
- 新增或修改函数必须有注释，至少包含：`功能说明`、`使用示例`。
- 注释描述当前真实行为、输入输出、错误语义和关键边界。
- 简单转发或小 helper 也要说明存在原因；无意义 helper 应删除或内联。
- 注释优先解释原因、约束、不变量、错误语义、资源生命周期和非显然取舍；不得复述代码表面动作。
- 注释不得用“临时兼容”“过时接口”等历史口径掩盖当前行为。

## 公开 API
- 新增、删除、重命名或修改公开 `API` 前，必须有用户确认来源。
- 公开 `API` 包括 `spec/API 列表`、函数签名、类方法签名、工具入口、脚本参数、include 接口和稳定错误语义。
- 不得把 helper、别名、兼容入口或测试便利函数写进 `API 列表`。
- 缺少用户确认时，只能写待确认项，不得实现或放行。

## 跨文件调用
- 当前文件之外，只允许调用对应 `spec` 和文件级 `API 列表` 中声明的公开 `API`。
- 非公开 helper 只能在当前文件内部使用。
- 不得通过包装转发、别名导入、反射、动态属性探测或字符串调用绕过公开 `API` 边界。
- 不得使用 `hasattr/getattr/callable(getattr(...))` 为上下文对象做运行时兼容分支；能力缺失应回到 `spec` 和公开 API 收口。
- 发现必须跨文件使用非公开 helper 时，先回到 `spec` 和架构裁定。

## 类型与错误
- 函数签名应使用明确类型；不得用 `object` 掩盖可枚举输入。
- 存在多种可接受输入时，必须用明确联合类型或多个具体类型写清。
- 宽类型只有在用户确认且 `spec` 写成公开合同时才允许。
- 稳定错误文本属于公开语义；修改前必须有用户确认来源。
- 错误类型、错误模块、错误消息和公开失败行为应与 `spec` 保持一致。

## 实现质量
- 避免过大函数、浅模块、无意义小函数、重复逻辑、过期注释和未使用代码。
- 复杂逻辑应拆到当前文件内清晰 helper；跨文件复用必须通过公开 API。
- helper 必须隐藏真实复杂度或复用明确知识；只转发、只改名、只包一层的 helper 应删除或内联。
- 公开接口保持小而明确，内部实现可以复杂但必须被命名、注释和测试约束住。
- 命名必须表达领域含义、单位、生命周期、边界和错误含义；不得使用含糊缩写或只能靠上下文猜测的名称。
- 重复规则、常量、错误语义、转换逻辑和边界判断必须收敛到单一真源；只有当抽象会明显降低理解成本时才抽取。
- 不做投机泛化；只为当前已确认公开行为设计最小可维护结构。
- 除装饰器实现所必需的闭包场景外，不得在函数体内部定义嵌套函数。
- 注意输入输出校验、资源释放、并发安全、性能影响和可维护性。
- 不得为了测试或短期验收写只服务当前 case 的伪实现。

## 测试要求
- 测试只验证公开 `API`。
- 不得为了测试方便直连跨文件非公开 helper。
- 修改实现后必须按实际 diff 反推 pytest、测试脚本或可运行本地脚本。
- `expectation` 单列为合同验收，不替代 diff 反推测试。

## 自检
- 文件级说明和 `API 列表` 是否同步。
- 新增函数注释、示例和关联 `spec/test` 是否准确。
- 是否没有跨文件调用非公开 `API`。
- 是否没有新增未确认公开入口或兼容分支。
- 是否没有增加不必要复杂度、浅抽象、重复知识、投机泛化或含糊命名。
- 注释是否解释约束、原因和不变量，而不是复述代码。
- 测试是否会在实现错误时失败。
