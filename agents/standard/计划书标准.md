# 计划书标准

## 文件说明
- 功能说明：定义本仓库计划书的制作方法、结构要求、自检项与架构师互评口径。
- 使用示例：
  - 输入粗草稿：[`analysis .plan.md`](../../analysis%20.plan.md)
  - 产出计划书：`ARCHITECTURE/plan/<topic>_green_plan.md`
  - 搭配模板：[`agents/standard/计划书模板.md`](计划书模板.md)
  - 完成态参考：[`agents/standard/计划书完成样板.md`](计划书完成样板.md)

## 本仓库专项要求

- 计划书默认不是抽象路线图，而是面向当前仓库的可执行设计文档。
- 计划书必须显式写出本次要收口的公开 `spec / API`，不能只写“支持某能力”。
- 计划书凡涉及新增、删除、重命名或修改公开 `API`，必须先取得用户明确同意，并在正文写明用户确认来源；未确认前只能列为待确认项，不得写成最终设计或建任务。
- 计划书必须给执行人代码示例，让执行人直接看到“最后接口应长什么样、如何调用、输出应是什么”。
- 计划书必须写出对应的验收资产，验收资产可以是 `expectation`、`pytest`、单文件脚本，或“由执行者在该阶段补齐的测试/校验入口”。
- 若计划目标涉及公开行为、IR、源码输出、工具输出或稳定错误文本，优先用 `expectation` 锁定；若暂不适合写 `expectation`，也必须明确由哪类测试或脚本收口。
- 计划书中的“验收必过项目”只是最低验证集合；执行人与审查人仍必须按实际 diff 反推测试，不能因为计划书列出的命令通过就跳过改动文件对应测试；`expectation` 属于合同验收资产，不计入 diff 反推测试。
- 一个计划书只对应一个计划级大任务；计划书内部必须拆成多个可执行的计划内小任务卡，但这些小任务卡不再拆成独立 `TODO` 状态。
- 计划级大任务按 `execute（合并原 spec/build） -> review -> 架构复核 / 终验 -> merge/归档` 接续；`execute` 必须一次完成计划书内全部小任务卡的规格、实现、测试与验收闭环。
- 计划书终验、复验或终验修复复核的通过与否，只基于计划正文当前保留的预期功能、公开 `API` 与当前验收资产。
- `done_plan`、历史记录、对齐记录、归档说明、旧阶段编号与文字同步问题，若不影响当前预期功能、公开 `API` 或当前验收资产，不得单独作为“不通过”理由。
- 计划书默认不要为 `execute` 阶段逐文件限制实现侧只能修改哪些代码；除 `expectation / 计划书 / agents 标准文档 / immutable 内容` 等显式禁止面外，执行人默认可自由修改与任务目标直接相关的 `spec`、实现、测试和必要辅助文件。
- 计划书应按“模块范围”约束实现边界，而不是按“只允许改文件 A/B/C”约束实现边界；只有涉及 `immutable`、用户显式禁止面或计划资产本身时，才逐文件写禁止修改面。
- 计划书中的任务目标、小任务目标、修复目标都必须写成执行人可直接落地的动作语句，例如“添加 `xxx` 接口”“为 `xxx` 函数增加 `yyy` 功能”“跑通 `pytest ...` / `python3 -m expectation...`”；不得写成“继续推进”“按当前口径收口”“整理一下结构”这类泛目标。
- 除非用户明确要求其他起步方式，否则计划级大任务默认从 `execute` 类型起步；如果草案里仍写 `spec` 或 `build` 作为计划任务类型，互评前应先改正。
- 计划书若使用 `expectation` 作为合同，必须显式写清“合同真源顺序”；默认推荐顺序为：`expectation > spec > test 文字说明 > 当前实现`。
- 架构师在新建、修改或补件 `expectation` 时，必须先核对路径、入口、依赖、示例可执行性与目标输出；不得把未运行、未核实的 `expectation` 直接下发给执行链。
- 计划书只要存在不确定项目，必须先向用户确认；未确认前不得自行补假设，也不得把主观推断写成既定方案。
- 公开 `API` 的新增、删除、重命名、参数顺序、返回值、默认值、公开错误文本和工具参数变化一律视为必须用户确认的事项。
- 只做计划书时，默认必须在推进前完成不少于 `3` 个对象的讨论记录；这是强规则，不需要用户额外点名。
- 计划书阶段的讨论对象默认从 [`agents/codex-multi-agents/agents-lists.md`](../codex-multi-agents/agents-lists.md) 选择，并通过 `-talk` 留下可追溯记录；除非用户明确要求改用其他对象或其他记录方式，否则不得只在会话里口头概述“已讨论”。
- 计划书阶段凡是存在争议、冲突、口径不一致或不确定事项，一律由用户确认；未收到用户确认前，不得擅自做主。
- 计划书起草顺序固定为：先完成不少于 `3` 个对象的讨论，再把仍影响方案、范围、依赖、完成态或验收口径的事项提交给用户拍板，收到用户决定后才开始正式写计划书正文；不得先写正式计划书，再回补讨论记录或用户确认。

## 适用范围

- 适用于 [`ARCHITECTURE/plan`](../../ARCHITECTURE/plan) 下的专题计划书。
- 适用于以下几类计划：
  - DSL 前端语义、IR 生成、pass、codegen、工具链专题；
  - include / API / target / 验收资产专题；
  - 单一重构目标的阶段化推进计划。
- 不适用于：
  - 单次实现记录；
  - 审查报告；
  - 任务记录；
  - `spec` 正文。

## 制作流程

### 1. 计划阶段只做计划

- 写计划书时，只允许收口目标、边界、样例、输出、计划内小任务、待确认项与复核摘要。
- 不在计划书任务里顺手改实现，不把“边想边试出来的代码”混进计划阶段。
- 计划阶段可修改的内容，只应包括计划书正文、评审摘要、必要的批注修订文本，以及直接配套的模板/标准文档。
- 若已经进入实现、测试、脚本或功能文件修改，应视为进入执行阶段，不再属于“写计划书”。

### 2. 先讨论，再请用户拍板，再写正文

- 计划书正文不是讨论起点；讨论记录和用户拍板应先于正式正文。
- 至少先与 `3` 个对象完成讨论，并记录：
  - 当前目标与范围理解；
  - 公开 `API` 与非公开 helper 边界；
  - 是否存在影响计划内小任务、依赖顺序、完成态或验收口径的分歧。
- 讨论对象默认应来自 [`agents/codex-multi-agents/agents-lists.md`](../codex-multi-agents/agents-lists.md)，并通过 `-talk` 发送；若因用户明确要求改用其他对象或其他方式，必须在计划书正文写明例外原因。
- 讨论后，如果仍有会影响方案、范围、依赖、完成态或验收口径的事项，必须先整理成“选项 + 差异 + 影响 + 推荐项”交给用户决定。
- 在用户未拍板前，只允许整理输入、记录分歧和补讨论摘要；不得把计划书写成可下发的正式版。

### 3. 先理解目标，再开始拆计划内小任务

- 计划不是从“有哪些文件要改”开始，而是从“完成时必须看到什么结果”开始。
- 用户草稿、现有实现、现有测试、现有验收资产要先对齐，再进入计划内小任务设计。

### 4. 计划必须保留争议与待确认项

- 如果方案存在多种合理写法，计划书里必须显式写出差异、影响和推荐项。
- 不清楚的地方不能靠默认假设补齐，必须变成待确认项。
- 只要待确认项仍影响方案、计划内小任务或验收口径，就必须先向用户确认；未收到确认前，不得自行补全为默认方案。

### 5. 计划必须可被执行人直接照做

- 每个计划内小任务都必须自包含；执行人只读对应小任务卡，也能判断为什么做、做什么、不做什么、怎么验收、记录写到哪里。
- 计划内小任务正文必须写清最小功能闭环，避免只交付孤立 import、临时报错修复或没有对应测试的局部改动。
- 计划内小任务正文必须写清 `执行步骤`；步骤要按实际落地顺序描述具体动作，例如“更新公开 API 列表”“实现入口行为”“补测试与残留扫描”，不得只写“按计划实现”。
- 对执行阶段，计划书应优先写“禁止修改面”和“合同真源”，不要把实现侧写成过窄白名单；若确需说明修改范围，也以这两项为主，不强制单列“可改文件”。
- 计划内小任务正文若需限制实现侧范围，应优先写“目标模块 / 子系统范围”，例如“限定在 `kernel_gen/passes/tile/**` 模块内收口”；不得把正常实现任务写成逐文件白名单，导致执行人为满足闭环仍需反复申请扩范围。

### 6. 计划必须可被管理员直接推进

- 管理员不应先读代码才能分发任务。
- 因此计划书中的阶段信息、工作树、记录文件、任务目标、通过条件，都必须一眼可用。

### 7. 整理用户输入

- 先把用户原始诉求收成四句话：
  - 目标是什么；
  - 不做什么；
  - 当前痛点是什么；
  - 完成后用户最想看到什么例子。
- 如果用户给的是草稿、示例代码、失败 case 或零散口头要求，先整理成一份“输入摘要”再继续。

### 8. 补当前基线

- 必须核对：
  - 当前 `spec` 如何定义；
  - 当前公开 `API` 是什么；
  - 当前实现在哪里；
  - 当前测试和验收资产覆盖到哪里；
  - 当前失败点是什么。
- “当前基线”要写成事实，不写愿景。

### 9. 确认待决策项

- 以下内容只要有一项不清楚，就先整理为待确认项：
  - 公开接口长什么样；
  - 具体要收口哪个 `spec / API`；
  - 输入输出样例如何写；
  - 验收资产如何命名、放在哪里、由谁补齐；
  - 是否保留旧行为；
  - 限定在哪个模块或子系统内收口；
  - 哪些路径属于禁止修改面；
  - 哪些失败路径要显式保留。
- 待确认项必须逐条写清“选项、差异、影响、推荐项”。
- 如果 `expectation` 路径、入口、依赖和输出已经明确，则“实现与 expectation 冲突”不属于待确认项，默认由执行人收口实现，不再升级为架构问题。
- 若待决策项需要用户拍板，计划书正文必须明确写成“待用户确认”，在收到确认前不得自行补假设。
- 若待决策项涉及公开 `API` 变更，在收到用户确认前不得进入互评通过、建任务、通知管理员推进或交给执行人实现。
- 计划书正文或配套记录必须写明不少于 `3` 个讨论对象及其结果；对象数量不足 `3` 时不得推进。
- 讨论对象记录除姓名与结论摘要外，还必须写明 `-talk` 记录路径或等效留痕位置；仅写“已讨论”但无法追溯消息来源时，视为未满足要求。

### 10. 先收口完成态，再拆计划内小任务

- 先写“完成时外部能看到什么”，再拆 `S1 / S2 / S3 ...` 计划内小任务。
- 每个计划内小任务都要有独立产出，不能只写“为后续做准备”。
- 计划内小任务是计划级大任务内部的可执行任务卡，不再对应独立 `TODO` 状态；默认让整份计划在同一个任务内完成规格、实现与验收闭环。

### 11. 写方案比较与选型

- 至少回答三件事：
  - 为什么不选现有替代方案；
  - 本次选型的最短路径是什么；
  - 这个选型以后是否容易维护。
- 计划书不是讨论帖，最后必须有明确选型结论。

### 12. 写计划级任务与计划内小任务正文

- 计划级任务负责给管理员看全局、任务类型、工作树和记录文件。
- 计划内小任务正文负责给执行人看细节。
- 两部分缺一不可，但不要互相复制整段说明。
- 每个计划内小任务至少包含：上下文摘要、本小任务目标、非目标、禁止修改面 / 合同真源、最小功能闭环、执行步骤、预期输出、验收必过项目、Diff 反推要求、记录要求。
- `执行步骤` 必须是执行人可照做的有序动作；如果步骤缺失、过粗或只重复目标，计划书不得进入可下发状态。
- 计划级 `任务目标` 与每个计划内小任务的 `小任务目标` 都必须是“可执行动作 + 对象 + 验收结果”的写法；若执行人读完后仍无法直接判断该写哪个接口、改哪个行为、跑哪个测试或合同入口，视为计划书不合格。
- `验收必过项目` 只是最低集合；执行人与审查人仍必须按实际 diff 反推测试。

### 13. 互评后再进入推进

- 计划书作者先自检。
- 再由另一位架构师按“任务划分、设计合理性、后续维护性、文本清晰度”互评。
- 有修改项先改完，再补评审摘要，之后才进入管理员推进。
- 若用户确认尚未完成，或讨论对象记录数量不足 `3`，则不得进入互评完成态、建任务或通知管理员推进。
- 若讨论记录和用户拍板都尚未完成，则当前文档只能标记为“讨论提纲”或“待讨论版”，不得当作正式计划书使用。

### 14. 允许用批注方式修订计划书

- 互评或复审时，可以直接在计划书中使用 `CMT: ... ENDCMT` 批注问题、歧义和修改建议。
- 批注内容只用于计划文本修订，不用于实现说明、代码讨论或任务记录。
- 批注应聚焦：
  - 任务划分是否合理；
  - 设计是否合理；
  - 后续维护是否顺；
  - 文本是否清楚。
- 批注收口后，要把最终结论写回计划正文或评审摘要，不保留已处理完的批注块。

### 15. 终验不通过时回到计划级任务修复

- 计划书终验若结论为“不通过”，默认不归档，也不把问题留在会话里等待下次再说。
- 给出计划书终验、复验或终验修复复核结论的架构师，必须把检验报告或终验结论写回计划书正文；至少写清结论人、结论、验证基线、最小阻断项或通过摘要，不得只停留在会话或 `talk.log`。
- 终验、复验或终验修复复核一律从严：只要在计划正文当前保留的预期功能、公开 `API` 或当前验收资产范围内，还能明确指出任何一处可执行的缺口、收敛点、可维护性优化点或测试增强点，就不得给出“通过”。
- 必须先把最小阻断项写成明确修复范围；规格、实现、测试或验收缺口回到同一个计划级 `execute` 任务继续处理，审查记录缺口回到 `review` 处理。
- 管理员若在终验不通过后尚未看到修复去向，必须主动询问 `守护最好的爱莉希雅` 给出原计划级任务的接续类型、目标和记录位置；另一位架构师只补充阻断项、修复边界和验收重点，不再重复创建平行任务。
- 修复接续必须继续挂在原计划书下，任务目标直接指向“让原计划书通过”，而不是另起一份平行计划。
- 如果两位架构师误建了重复平行任务，只保留一个范围更清楚、依赖正确、`worktree` 明确的计划级任务继续推进，另一项在任务记录中标注为重复后停止继续推进。
- 修复完成后，必须回到原计划书重新终验；只有两位架构师都改口为“通过”，才允许通知管理员归档。

## 计划书结构标准

计划书建议固定为以下顺序：

### 1. 标题

- 文件名建议：`<topic>_green_plan.md`
- 标题和文件名保持同一主题，不要出现一个文件承载两个无关目标。

### 2. 文档信息

- 至少包含：
  - 创建者
  - 最后一次更改
  - 目标 `spec`
  - 目标 `API`
  - 目标 `test`
  - 目标 `验收资产`
  - 目标 `功能实现`
- 这一节只列与本计划直接相关的文件。

### 3. 计划级任务

- 使用表格统一记录：
  - 计划任务名
  - 任务类型
  - `worktree`
  - 记录文件
- 一个计划书只写一个计划级任务；禁止把 `S1 / S2 / S3` 计划内小任务拆成多个 `TODO` 状态。

### 4. 评审摘要

- 至少包含：
  - 评审结论
  - 评审人
  - 结论摘要
- 摘要不重复整份计划，只写关键共识。

### 4.1 终验 / 复验 / 修复复核记录

- 计划书终验、复验或终验修复复核的最终结论，必须直接写在计划书正文里，不得只停留在会话、`talk.log` 或口头消息。
- 历史记录、`done_plan`、对齐记录、归档说明与旧阶段编号，只在影响当前预期功能、公开 `API` 或当前验收资产时才进入最小阻断项；否则只能作为背景说明，不得单独阻断通过。
- 至少包含：
  - `结论人`
  - `结论`
  - `验证基线`
  - `最小阻断项或通过摘要`
- 若结论为“不通过”，正文还应写清：
  - `是否已安排原计划级任务回到 execute/review`
  - `修复接续负责人`
  - `另一位架构师补充的阻断项 / 修复边界 / 验收重点`
- 若尚未安排修复接续，管理员按通用规则询问 `守护最好的爱莉希雅` 给出原计划级任务接续口径；计划书正文不得改写成由管理员或两位架构师同时各建一份平行任务。

### 5. 计划目标

- 用 3 到 6 条短句写清：
  - 要新增或收口什么能力；
  - 对外公开入口是什么；
  - 公开 `API` 最终长什么样；
  - 计划产出给谁用；
  - 计划完成后减少了什么重复或歧义。

### 6. 当前基线

- 必须写出“今天仓库是什么状态”。
- 建议按四类写：
  - 当前公开合同；
  - 当前公开 API；
  - 当前实现入口；
  - 当前测试与验收资产；
  - 当前缺口或失败点。

### 7. 方案比较与选型

- 至少回答：
  - 不采用什么方案；
  - 原因是什么；
  - 本次采用什么方案；
  - 这个方案的最小公开接口是什么。

### 8. 公开 API 设计

- 这一节专门写本次计划要收口的公开接口。
- 至少包含：
  - 函数 / 类 / 工具入口名称；
  - 参数顺序；
  - 参数类型；
  - 返回值；
  - 一段最小调用示例。
- 如果是多层接口，必须写清哪一层是对外公开口径，哪一层只是内部辅助入口。

### 9. 完成态定义

- 这一节只写“最终要看到什么结果”，不写阶段过程。
- 必须是外部可判定内容，例如：
  - 某个接口签名固定；
  - 某段源码必须出现；
  - 某个验收输入必须得到指定输出；
  - 某组测试、脚本或 expectation 必须通过。

### 10. 验收设计

- 这一节专门说明本次计划如何锁定最终效果。
- 至少包含：
  - 要沿用或新增哪些验收资产；
  - 每个验收资产覆盖哪类行为；
  - 输入样例是什么；
  - 输出要锁定什么片段或结果；
  - 哪些命令是最终必须通过的。
- 若当前仓库还没有现成的验收资产，计划书必须直接说明由执行者在该阶段补齐什么测试或脚本，而不是只写“后续补”。
- 若计划书后续需要终验、复验或终验修复复核，验收设计里必须预留“验证基线”口径：先确认主目录代码是否已对齐到最新主线；若没有，则明确改用哪个最新远端现场，再说明应在什么现场复核，例如具体 `worktree`、分支 / `commit`、执行目录，或“主目录已同步到最新主线 / 最新远端现场”的说明。
- 若根目录旧主仓工作目录与上述验证基线不一致，只能记为现场状态差异；除非相同缺口在已写明的验证基线里也复现，否则不得把旧工作目录状态单独写成功能阻断或“不通过”依据。
- 验收设计必须写明：计划书最低命令不替代执行人与审查人按实际 diff 补跑对应测试；任务日志要记录 `Diff 反推自测` / `Diff 反推审查`；`expectation` 合同验收要另列。
- 计划书终验、复验或终验修复复核时，架构师必须在最新同步现场运行与本轮改动有关的 `expectation` 合同验收；默认先列出相关 `expectation` 入口，再逐项运行。若相关 `expectation` 未运行或存在失败，不得给出通过结论；只有用户明确要求时才运行全量 `expectation`。

```bash
while IFS= read -r mod; do
  python3 -m "$mod"
done < coverage/final/related_expectation_modules.txt
```

如果用户明确要求全量 `expectation`，再使用下面的全量入口：

```bash
while IFS= read -r main; do
  mod=${main#./}
  mod=${mod%/__main__.py}
  mod=${mod//\//.}
  python3 -m "$mod"
done < <(find expectation -name __main__.py -print | sort)
```

### 11. 计划内小任务

- 每个计划内小任务建议固定包含以下小节：
  - `小任务目标`
  - `目标 spec / API`
  - `禁止修改面 / 合同真源`
  - `最小功能闭环`
  - `执行步骤`
  - `预期示例代码`
  - `预期输出`
  - `目标验收资产`
  - `验收必过项目`
- 若小任务内部仍然很大，可以再细分为 `S2A / S2B`，但只在确有必要时使用；这些编号仍然不是独立 `TODO` 状态。

### 11.1 修复接续怎么写

- 若某个计划内小任务或整份计划在终验时未通过，应在原计划级任务下补“修复接续”记录，不再新增平行计划子任务。
- 修复接续正文至少写清：
  - 修复的阻断项；
  - 只允许收口哪些文件或行为；
  - 修完后要回到原计划验证哪些资产；
  - 不在本次修复接续处理的内容。
- 修复接续的任务目标必须可直接映射为原计划级任务的 `execute` 或 `review` 类型切换，不写模糊表述；推荐直接写“修复 `xxx` 失败，使 `pytest ...` / `python3 -m expectation...` 通过”。
- 若修复接续来自终验不通过，阻断项必须来自已写明的最新同步现场；不得只因为根目录旧主仓工作目录较旧，就把现场差异直接写成新的功能缺口。

### 12. 待确认项

- 只保留尚未决定的问题。
- 每项写法固定为：
  - 问题
  - 可选项
  - 差异
  - 推荐项

### 12.1 用户确认与协同约束

- 计划书只要存在不确定项目，就必须写清用户确认状态、未确认事项与确认结果。
- 未确认事项未收口前，不得自行补假设，也不得把该项改写成既定方案。
- 计划书正文或配套记录必须写清：
  - 协同约束；
  - 不少于 `3` 个讨论对象；
  - 讨论对象是否来自 `agents-lists.md`；
  - 讨论是否通过 `-talk` 留痕；
  - 对应记录路径或结果摘要。
- 对象数量不足 `3` 时，不得进入互评完成态、建任务或通知管理员推进。

### 13. 参考资料

- 只保留和本计划直接相关的资料。
- 引用外部资料时，优先说明“借鉴点是什么”，不要只贴链接。

## 计划内小任务要求

- 每个计划内小任务只能有一个直接产出中心，不要把多个主题拼在一起。
- 如果一个计划内小任务需要改很多文件，先判断是不是目标过大，而不是直接接受“大任务卡”。
- 每个计划内小任务都要能独立验收，并让管理员判断原计划级任务是否可以进入下一角色阶段。

## 样例与输出写法

### 预期示例代码怎么写

- 只写最小输入，不写大段背景代码。
- 尽量直接覆盖用户最关心的入口。
- 若计划目标是公开 API，示例代码必须直接使用该 API，而不是绕到内部 helper。
- 示例代码要让执行人一眼看出参数顺序、参数类型和返回结果。
- 样例优先覆盖：
  - 静态 case
  - 动态 case
  - 标量混合 case
  - 失败 case

### 预期输出怎么写

- 优先写最小、稳定、可直接 grep 的输出。
- 对源码类输出，优先写必须出现的关键片段。
- 对 IR 类输出，优先写固定结构、关键 op、关键 attribute。
- 对错误输出，写稳定短语，不写不稳定上下文。
- 如果最终行为主要由 `expectation` 锁定，预期输出要与 `expectation` 中的检查文本保持同一口径。
- 若 `expectation` 已锁定稳定输出，而当前实现未满足该输出，默认由执行人修改实现与测试对齐；不要把这类问题再写成“需架构确认”。

### 验收资产怎么写进计划书

- 计划书里至少要回答四个问题：
  - 验收资产是沿用、新增，还是由执行者补齐；
  - 验收资产路径或命令是什么；
  - 输入样例是什么；
  - 最终要锁定什么输出。
- 如果目标是公开行为收口，计划书不能只写“补测试”，要写清执行者需要补哪类测试或脚本。
- 如果目标是源码、IR、工具输出、报错信息，优先用 `expectation` 锁定，不只靠单元测试断言。

### 验收必过项目怎么写

- 只写可直接执行或可直接运行的项目：
  - `pytest`
  - `expectation` 脚本
  - 单文件工具入口
- 先写最小范围，再写目录范围。
- 命令顺序应体现从局部到整体的收口过程。
- 计划内小任务正文中应补一句：除本节列出的最低命令外，执行人与审查人还需按实际 diff 反推测试，并把结果写入任务日志；`expectation` 合同验收单列，不算 diff 反推测试。

## 计划内小任务标准

### 好的小任务

- 一个计划内小任务只围绕一个公开入口或一类行为。
- 计划级任务信息描述整份计划的直接产出；计划内小任务描述顺序、边界和验收。
- 执行人看完任务目标与计划内小任务正文，就知道自己要改哪些范围、做到什么结果、怎样自查。
- 执行人看完计划后，能直接根据计划里的 API 示例和验收资产判断自己要做到什么。

### 不好的小任务

- 同时把接口重写、实现重构、工具扩展、验收资产全部塞进一个计划内小任务。
- 小任务目标写成“完善整体能力”“补齐所有细节”。
- 只写“按 spec 实现”，但没有样例、输出、通过条件。

## 自检清单

### 必备字段 checklist

- [ ] 标题与文件名同主题。
- [ ] 文档信息已写全：创建者、最后一次更改、目标 `spec`、目标 `API`、目标 `test`、目标 `验收资产`、目标 `功能实现`。
- [ ] 计划级任务已列出计划任务名、任务类型、`worktree`、记录文件。
- [ ] 评审摘要已写结论、评审人、结论摘要。
- [ ] 输入摘要或等价内容已说明：目标、不做什么、当前痛点、完成后最想看到的例子。
- [ ] 计划目标、当前基线、方案比较与选型、公开 API 设计、完成态定义、验收设计、计划内小任务、待确认项、参考资料齐全。
- [ ] 每个计划内小任务都含“小任务目标 / 目标 spec / API / 禁止修改面 / 合同真源（如需）/ 最小功能闭环 / 执行步骤 / 预期示例代码 / 预期输出 / 目标验收资产 / 验收必过项目”。
- [ ] 必过命令可直接运行，不是空泛描述。

写完计划书后，至少逐条检查：

- 目标是否能用外部结果判断，而不是只能读实现才能判断。
- 当前基线是否是仓库事实，而不是推测。
- 是否显式写出了本次收口的 `spec / API / 验收资产`。
- 是否写清了 `expectation` 是否为合同真源，以及真源顺序是什么。
- 每个计划内小任务是否只有一个直接产出中心。
- 每个计划内小任务是否都给了最小输入样例和最小输出样例。
- 示例代码是否已经足够表达公开接口与参数顺序。
- 验收资产是否已经指定到路径、命令或补建要求，而不是只写“补测试”。
- 验收项目是否能直接运行。
- 若计划书引用终验、复验或终验修复复核现场，是否已经先说明主目录代码是否对齐最新主线或本轮指定的最新远端现场，并写清最新同步现场与验证基线，而不是只写“当前主仓”。
- 是否把根目录旧主仓工作目录较旧这一现象只记为现场状态差异，而没有单独写成功能阻断。
- 是否把实现侧边界写成了过窄的逐文件白名单；若是，应改成“禁止修改面 + 合同真源”。
- 计划中是否还留有作者自己未说明白的接口或顺序。
- 如果不照抄计划正文，自己是否仍能用几句话复述清楚：目标、边界、公开 API、最小样例、验收资产。
- 是否明确写出了不做什么。
- 是否把实现细节、历史迁移过程、个人心路历程混进了计划正文。
- 公开 `API`、参数顺序、默认值、异常语义和边界是否合理，是否存在功能重复、职责重叠或维护成本过高的设计。
- 新 `API` 或行为变更是否说明兼容性、破坏性变更、弃用路径、默认值迁移和旧调用残留清理。
- 性能、可靠性、安全、并发、资源生命周期和失败恢复是否在需要时被纳入方案或明确不适用。
- 示例是否足够短、准确、机械可判，执行人是否能一眼看懂输入、输出和完成标准。
- 计划内小任务是否给了执行人足够上下文，是否避免“只看任务摘要就开工”导致误做。
- 计划内小任务是否过大、混入多个功能或复杂度过高；如果是，是否已拆分或写明串并行依赖。
- 验收是否既包含正向主路径，也覆盖负例、边界、回归和残留扫描；若依赖环境例外，是否写清例外条件。
- 方案是否已经比较过主要取舍；若还有更优或更稳方案，是否写入方案比较或待确认项。
- 是否已经主动找出所有不完美、可修改、可降低风险的点，而不是只写到“看起来能推进”。

## 互评口径

架构师互评建议优先看以下四项：

### 1. 任务划分是否合理

- 是否过大；
- 是否能并行；
- 是否会让执行人同时承担两个主题。

### 2. 设计是否合理

- 公开接口是否清楚；
- 公开 API 是否已经收口到参数、返回值与调用样例；
- 公开 API 变更是否已经写明用户确认来源；缺失时不得通过。
- 方案是否绕远；
- 是否误把当前实现偶然性写成长期合同。

### 3. 后续维护是否顺

- 是否把真正稳定的公开口径写清楚；
- 是否把内部辅助流程错写成公开接口；
- 后续增加新 case 时是否容易沿用。

### 4. 文本是否清楚

- 样例是否足够直接；
- 验收资产路径、命令或补建要求是否写清楚；
- 输出是否可判断；
- 是否存在多处表述互相冲突。
- `expectation` 是否已经被架构师实际核对可运行，而不是只停留在文字设想。

## 推荐做法与不推荐做法

### 推荐做法

- 先写完成态，再写计划内小任务。
- 先定公开 API，再展开实现小任务。
- 先写样例，再写抽象说明。
- 先把验收资产与最终输出锁定，再拆执行步骤。
- 先写仓库事实，再写调整方向。
- 对量化目标先问清是必须达到还是优化方向。
- 对多方案问题保留明确选型记录。

### 不推荐做法

- 用“支持更多场景”“增强可维护性”这类空话代替完成态。
- 用“看实现决定”“细节自定”代替接口定义。
- 只写“补测试”，不写验收资产和预期输出。
- 在计划书里直接写实现代码块，导致计划与实现混在一起。
- 只列文件名，不写样例和输出。
- 跳过当前基线，直接从理想状态开始写。
