# 计划书标准

## 文件说明
- 功能说明：定义本仓库计划书的制作方法、结构要求、自检项与架构师互评口径。
- 使用示例：
  - 输入粗草稿：[`analysis .plan.md`](../../analysis%20.plan.md)
  - 产出计划书：`ARCHITECTURE/plan/<topic>_green_plan.md`
  - 搭配模板：[`agents/standard/计划书模板.md`](计划书模板.md)
  - 完成态参考：[`agents/standard/计划书完成样板.md`](计划书完成样板.md)

## 本仓库专项要求

- 计划书默认不是抽象路线图，而是面向当前仓库的可执行设计文档。
- 计划书必须显式写出本次要收口的公开 `spec / API`，不能只写“支持某能力”。
- 计划书必须给执行人代码示例，让执行人直接看到“最后接口应长什么样、如何调用、输出应是什么”。
- 计划书必须写出对应的验收资产，验收资产可以是 `expectation`、`pytest`、单文件脚本，或“由执行者在该阶段补齐的测试/校验入口”。
- 若计划目标涉及公开行为、IR、源码输出、工具输出或稳定错误文本，优先用 `expectation` 锁定；若暂不适合写 `expectation`，也必须明确由哪类测试或脚本收口。
- 计划书默认不要为 `build` 阶段逐文件限制实现侧只能修改哪些代码；除 `expectation / spec / 计划书 / agents 标准文档 / immutable 内容` 等显式禁止面外，执行人默认可自由修改与任务目标直接相关的实现、测试和必要辅助文件。
- 计划书若使用 `expectation` 作为合同，必须显式写清“合同真源顺序”；默认推荐顺序为：`expectation > spec > test 文字说明 > 当前实现`。
- 架构师在新建、修改或补件 `expectation` 时，必须先核对路径、入口、依赖、示例可执行性与目标输出；不得把未运行、未核实的 `expectation` 直接下发给执行链。
- 计划书只要存在不确定项目，必须先向用户确认；未确认前不得自行补假设，也不得把主观推断写成既定方案。
- 若用户明确要求至少询问 `3` 人，计划书阶段必须显式写明该协同约束，并在推进前完成不少于 `3` 个对象的询问记录。

## 适用范围

- 适用于 [`ARCHITECTURE/plan`](../../ARCHITECTURE/plan) 下的专题计划书。
- 适用于以下几类计划：
  - DSL 前端语义、IR 生成、pass、codegen、工具链专题；
  - include / API / target / 验收资产专题；
  - 单一重构目标的阶段化推进计划。
- 不适用于：
  - 单次实现记录；
  - 审查报告；
  - 任务记录；
  - `spec` 正文。

## 制作流程

### 1. 计划阶段只做计划

- 写计划书时，只允许收口目标、边界、样例、输出、任务拆分、待确认项与复核摘要。
- 不在计划书任务里顺手改实现，不把“边想边试出来的代码”混进计划阶段。
- 计划阶段可修改的内容，只应包括计划书正文、评审摘要、必要的批注修订文本，以及直接配套的模板/标准文档。
- 若已经进入实现、测试、脚本或功能文件修改，应视为进入执行阶段，不再属于“写计划书”。

### 2. 先理解目标，再开始拆任务

- 计划不是从“有哪些文件要改”开始，而是从“完成时必须看到什么结果”开始。
- 用户草稿、现有实现、现有测试、现有验收资产要先对齐，再进入任务拆分。

### 3. 计划必须保留争议与待确认项

- 如果方案存在多种合理写法，计划书里必须显式写出差异、影响和推荐项。
- 不清楚的地方不能靠默认假设补齐，必须变成待确认项。
- 只要待确认项仍影响方案、任务拆分或验收口径，就必须先向用户确认；未收到确认前，不得自行补全为默认方案。

### 4. 计划必须可被执行人直接照做

- 每个阶段都要写清：
  - 当前阶段要改什么；
  - 完成后代码或输出应长什么样；
  - 用什么样例可以直接判断结果；
  - 哪些内容不在当前阶段处理。
- 对执行阶段，计划书应优先写“禁止修改面”和“合同真源”，不要把实现侧写成过窄白名单；若确需说明修改范围，也以这两项为主，不强制单列“可改文件”。

### 5. 计划必须可被管理员直接推进

- 管理员不应先读代码才能分发任务。
- 因此计划书中的阶段信息、工作树、记录文件、任务目标、通过条件，都必须一眼可用。

### 6. 整理用户输入

- 先把用户原始诉求收成四句话：
  - 目标是什么；
  - 不做什么；
  - 当前痛点是什么；
  - 完成后用户最想看到什么例子。
- 如果用户给的是草稿、示例代码、失败 case 或零散口头要求，先整理成一份“输入摘要”再继续。

### 7. 补当前基线

- 必须核对：
  - 当前 `spec` 如何定义；
  - 当前公开 `API` 是什么；
  - 当前实现在哪里；
  - 当前测试和验收资产覆盖到哪里；
  - 当前失败点是什么。
- “当前基线”要写成事实，不写愿景。

### 8. 确认待决策项

- 以下内容只要有一项不清楚，就先整理为待确认项：
  - 公开接口长什么样；
  - 具体要收口哪个 `spec / API`；
  - 输入输出样例如何写；
  - 验收资产如何命名、放在哪里、由谁补齐；
  - 是否保留旧行为；
  - 哪些文件可以改；
  - 哪些失败路径要显式保留。
- 待确认项必须逐条写清“选项、差异、影响、推荐项”。
- 如果 `expectation` 路径、入口、依赖和输出已经明确，则“实现与 expectation 冲突”不属于待确认项，默认由执行人收口实现，不再升级为架构问题。
- 若待决策项需要用户拍板，计划书正文必须明确写成“待用户确认”，在收到确认前不得自行补假设。
- 若用户要求至少询问 `3` 人，计划书正文或配套记录必须写明协同约束、询问对象与询问结果；对象数量不足 `3` 时不得推进。

### 9. 先收口完成态，再拆阶段

- 先写“完成时外部能看到什么”，再拆 `S1 / S2 / S3 ...`。
- 每个阶段都要有独立产出，不能只写“为后续做准备”。
- 默认把阶段写成“收口任务”，即一个阶段结束后应形成完整的规格、实现与验收闭环；只有在范围非常明确时，才拆成更细的专题阶段。

### 10. 写方案比较与选型

- 至少回答三件事：
  - 为什么不选现有替代方案；
  - 本次选型的最短路径是什么；
  - 这个选型以后是否容易维护。
- 计划书不是讨论帖，最后必须有明确选型结论。

### 11. 写任务清单与阶段正文

- 任务清单负责给管理员看全局与任务串联关系。
- 阶段正文负责给执行人看细节。
- 两部分缺一不可。

### 12. 互评后再进入推进

- 计划书作者先自检。
- 再由另一位架构师按“任务划分、设计合理性、后续维护性、文本清晰度”互评。
- 有修改项先改完，再补评审摘要，之后才进入管理员推进。
- 若用户确认尚未完成，或用户要求的多对象询问记录数量不足，则不得进入互评完成态、建任务或通知管理员推进。

### 13. 允许用批注方式修订计划书

- 互评或复审时，可以直接在计划书中使用 `CMT: ... ENDCMT` 批注问题、歧义和修改建议。
- 批注内容只用于计划文本修订，不用于实现说明、代码讨论或任务记录。
- 批注应聚焦：
  - 任务划分是否合理；
  - 设计是否合理；
  - 后续维护是否顺；
  - 文本是否清楚。
- 批注收口后，要把最终结论写回计划正文或评审摘要，不保留已处理完的批注块。

### 14. 终验不通过时直接拆修复任务

- 计划书终验若结论为“不通过”，默认不归档，也不把问题留在会话里等待下次再说。
- 必须先把最小阻断项写成明确修复范围，再围绕这些阻断项创建修复任务。
- 默认由“给出最终不通过结论的那位架构师”创建修复任务；另一位架构师只补充阻断项、修复边界和验收重点，不再重复建同一任务。
- 修复任务要继续挂在原计划书下，任务目标直接指向“让原计划书通过”，而不是另起一份平行计划。
- 如果两位架构师误建了同一修复任务，只保留一个任务继续推进；保留项以范围更清楚、依赖正确、`worktree` 明确者为准，另一项在任务记录中标注为重复任务。
- 修复任务完成后，必须回到原计划书重新终验；只有两位架构师都改口为“通过”，才允许通知管理员归档。

## 计划书结构标准

计划书建议固定为以下顺序：

### 1. 标题

- 文件名建议：`<topic>_green_plan.md`
- 标题和文件名保持同一主题，不要出现一个文件承载两个无关目标。

### 2. 文档信息

- 至少包含：
  - 创建者
  - 最后一次更改
  - 目标 `spec`
  - 目标 `API`
  - 目标 `test`
  - 目标 `验收资产`
  - 目标 `功能实现`
- 这一节只列与本计划直接相关的文件。

### 3. 任务清单

- 使用表格统一记录：
  - 任务编号
  - 前置任务
  - `worktree`
  - 记录文件
- 任务编号建议从 `S1` 开始，保持单层编号。

### 4. 评审摘要

- 至少包含：
  - 评审结论
  - 评审人
  - 结论摘要
- 摘要不重复整份计划，只写关键共识。

### 5. 计划目标

- 用 3 到 6 条短句写清：
  - 要新增或收口什么能力；
  - 对外公开入口是什么；
  - 公开 `API` 最终长什么样；
  - 计划产出给谁用；
  - 计划完成后减少了什么重复或歧义。

### 6. 当前基线

- 必须写出“今天仓库是什么状态”。
- 建议按四类写：
  - 当前公开合同；
  - 当前公开 API；
  - 当前实现入口；
  - 当前测试与验收资产；
  - 当前缺口或失败点。

### 7. 方案比较与选型

- 至少回答：
  - 不采用什么方案；
  - 原因是什么；
  - 本次采用什么方案；
  - 这个方案的最小公开接口是什么。

### 8. 公开 API 设计

- 这一节专门写本次计划要收口的公开接口。
- 至少包含：
  - 函数 / 类 / 工具入口名称；
  - 参数顺序；
  - 参数类型；
  - 返回值；
  - 一段最小调用示例。
- 如果是多层接口，必须写清哪一层是对外公开口径，哪一层只是内部辅助入口。

### 9. 完成态定义

- 这一节只写“最终要看到什么结果”，不写阶段过程。
- 必须是外部可判定内容，例如：
  - 某个接口签名固定；
  - 某段源码必须出现；
  - 某个验收输入必须得到指定输出；
  - 某组测试、脚本或 expectation 必须通过。

### 10. 验收设计

- 这一节专门说明本次计划如何锁定最终效果。
- 至少包含：
  - 要沿用或新增哪些验收资产；
  - 每个验收资产覆盖哪类行为；
  - 输入样例是什么；
  - 输出要锁定什么片段或结果；
  - 哪些命令是最终必须通过的。
- 若当前仓库还没有现成的验收资产，计划书必须直接说明由执行者在该阶段补齐什么测试或脚本，而不是只写“后续补”。
- 若计划书后续需要终验、复验或终验修复复核，验收设计里必须预留“验证基线”口径：明确说明应在什么最新同步现场复核，例如具体 `worktree`、分支 / `commit`、执行目录，或“已同步到最新主线的任务环境”。
- 若根目录旧主仓工作目录与上述验证基线不一致，只能记为现场状态差异；除非相同缺口在已写明的验证基线里也复现，否则不得把旧工作目录状态单独写成功能阻断或“不通过”依据。

### 11. 阶段拆分

- 每个阶段建议固定包含以下小节：
  - `阶段目标`
  - `目标 spec / API`
  - `禁止修改面 / 合同真源`
  - `预期示例代码`
  - `预期输出`
  - `目标验收资产`
  - `验收必过项目`
  - `任务新建建议`
- 若阶段内部仍然很大，可以再细分为 `S2A / S2B`，但只在确有必要时使用。

### 11.1 修复阶段怎么写

- 若某阶段或整份计划在终验时未通过，可直接补一个最小修复阶段，例如 `S6` 或 `S6-fix`。
- 修复阶段正文至少写清：
  - 修复的阻断项；
  - 只允许收口哪些文件或行为；
  - 修完后要回到原计划验证哪些资产；
  - 不在本修复阶段处理的内容。
- 修复阶段的任务目标必须可直接映射为一个新的 `build/review/merge` 链路，不写模糊表述。
- 若修复阶段来自终验不通过，阻断项必须来自已写明的最新同步现场；不得只因为根目录旧主仓工作目录较旧，就把现场差异直接写成新的功能缺口。

### 12. 待确认项

- 只保留尚未决定的问题。
- 每项写法固定为：
  - 问题
  - 可选项
  - 差异
  - 推荐项

### 12.1 用户确认与协同约束

- 计划书只要存在不确定项目，就必须写清用户确认状态、未确认事项与确认结果。
- 未确认事项未收口前，不得自行补假设，也不得把该项改写成既定方案。
- 若用户明确要求至少询问 `3` 人，计划书正文或配套记录必须写清：
  - 协同约束；
  - 询问对象；
  - 对应记录路径或结果摘要。
- 对象数量不足 `3` 时，不得进入互评完成态、建任务或通知管理员推进。

### 13. 参考资料

- 只保留和本计划直接相关的资料。
- 引用外部资料时，优先说明“借鉴点是什么”，不要只贴链接。

## 阶段拆分要求

- 每个阶段只能有一个直接产出中心，不要把多个主题拼在一起。
- 如果一个阶段需要改很多文件，先判断是不是目标过大，而不是直接接受“大阶段”。
- 每个阶段都要能独立结束，并让管理员判断“是否可以进入下一阶段”。

## 样例与输出写法

### 预期示例代码怎么写

- 只写最小输入，不写大段背景代码。
- 尽量直接覆盖用户最关心的入口。
- 若计划目标是公开 API，示例代码必须直接使用该 API，而不是绕到内部 helper。
- 示例代码要让执行人一眼看出参数顺序、参数类型和返回结果。
- 样例优先覆盖：
  - 静态 case
  - 动态 case
  - 标量混合 case
  - 失败 case

### 预期输出怎么写

- 优先写最小、稳定、可直接 grep 的输出。
- 对源码类输出，优先写必须出现的关键片段。
- 对 IR 类输出，优先写固定结构、关键 op、关键 attribute。
- 对错误输出，写稳定短语，不写不稳定上下文。
- 如果最终行为主要由 `expectation` 锁定，预期输出要与 `expectation` 中的检查文本保持同一口径。
- 若 `expectation` 已锁定稳定输出，而当前实现未满足该输出，默认由执行人修改实现与测试对齐；不要把这类问题再写成“需架构确认”。

### 验收资产怎么写进计划书

- 计划书里至少要回答四个问题：
  - 验收资产是沿用、新增，还是由执行者补齐；
  - 验收资产路径或命令是什么；
  - 输入样例是什么；
  - 最终要锁定什么输出。
- 如果目标是公开行为收口，计划书不能只写“补测试”，要写清执行者需要补哪类测试或脚本。
- 如果目标是源码、IR、工具输出、报错信息，优先用 `expectation` 锁定，不只靠单元测试断言。

### 验收必过项目怎么写

- 只写可直接执行或可直接运行的项目：
  - `pytest`
  - `expectation` 脚本
  - 单文件工具入口
- 先写最小范围，再写目录范围。
- 命令顺序应体现从局部到整体的收口过程。

## 任务拆分标准

### 好的拆分

- 一个阶段只围绕一个公开入口或一类行为。
- 一份任务信息只描述一个直接产出。
- 执行人看完任务目标与计划阶段正文，就知道自己要改哪些范围、做到什么结果、怎样自查。
- 执行人看完计划后，能直接根据计划里的 API 示例和验收资产判断自己要做到什么。

### 不好的拆分

- 同时把接口重写、实现重构、工具扩展、验收资产全部塞进一个阶段。
- 阶段目标写成“完善整体能力”“补齐所有细节”。
- 只写“按 spec 实现”，但没有样例、输出、通过条件。

## 自检清单

### 必备字段 checklist

- [ ] 标题与文件名同主题。
- [ ] 文档信息已写全：创建者、最后一次更改、目标 `spec`、目标 `API`、目标 `test`、目标 `验收资产`、目标 `功能实现`。
- [ ] 任务清单已列出阶段、前置任务、`worktree`、记录文件。
- [ ] 评审摘要已写结论、评审人、结论摘要。
- [ ] 输入摘要或等价内容已说明：目标、不做什么、当前痛点、完成后最想看到的例子。
- [ ] 计划目标、当前基线、方案比较与选型、公开 API 设计、完成态定义、验收设计、阶段拆分、待确认项、参考资料齐全。
- [ ] 每个阶段都含“阶段目标 / 目标 spec / API / 禁止修改面 / 合同真源（如需）/ 预期示例代码 / 预期输出 / 目标验收资产 / 验收必过项目 / 任务新建建议”。
- [ ] 必过命令可直接运行，不是空泛描述。

写完计划书后，至少逐条检查：

- 目标是否能用外部结果判断，而不是只能读实现才能判断。
- 当前基线是否是仓库事实，而不是推测。
- 是否显式写出了本次收口的 `spec / API / 验收资产`。
- 是否写清了 `expectation` 是否为合同真源，以及真源顺序是什么。
- 每个阶段是否只有一个直接产出中心。
- 每个阶段是否都给了最小输入样例和最小输出样例。
- 示例代码是否已经足够表达公开接口与参数顺序。
- 验收资产是否已经指定到路径、命令或补建要求，而不是只写“补测试”。
- 验收项目是否能直接运行。
- 若计划书引用终验、复验或终验修复复核现场，是否已经写清最新同步现场与验证基线，而不是只写“当前主仓”。
- 是否把根目录旧主仓工作目录较旧这一现象只记为现场状态差异，而没有单独写成功能阻断。
- 是否把实现侧边界写成了过窄的逐文件白名单；若是，应改成“禁止修改面 + 合同真源”。
- 计划中是否还留有作者自己未说明白的接口或顺序。
- 如果不照抄计划正文，自己是否仍能用几句话复述清楚：目标、边界、公开 API、最小样例、验收资产。
- 是否明确写出了不做什么。
- 是否把实现细节、历史迁移过程、个人心路历程混进了计划正文。

## 互评口径

架构师互评建议优先看以下四项：

### 1. 任务划分是否合理

- 是否过大；
- 是否能并行；
- 是否会让执行人同时承担两个主题。

### 2. 设计是否合理

- 公开接口是否清楚；
- 公开 API 是否已经收口到参数、返回值与调用样例；
- 方案是否绕远；
- 是否误把当前实现偶然性写成长期合同。

### 3. 后续维护是否顺

- 是否把真正稳定的公开口径写清楚；
- 是否把内部辅助流程错写成公开接口；
- 后续增加新 case 时是否容易沿用。

### 4. 文本是否清楚

- 样例是否足够直接；
- 验收资产路径、命令或补建要求是否写清楚；
- 输出是否可判断；
- 是否存在多处表述互相冲突。
- `expectation` 是否已经被架构师实际核对可运行，而不是只停留在文字设想。

## 推荐做法与不推荐做法

### 推荐做法

- 先写完成态，再写阶段。
- 先定公开 API，再展开实现阶段。
- 先写样例，再写抽象说明。
- 先把验收资产与最终输出锁定，再拆执行步骤。
- 先写仓库事实，再写调整方向。
- 对量化目标先问清是必须达到还是优化方向。
- 对多方案问题保留明确选型记录。

### 不推荐做法

- 用“支持更多场景”“增强可维护性”这类空话代替完成态。
- 用“看实现决定”“细节自定”代替接口定义。
- 只写“补测试”，不写验收资产和预期输出。
- 在计划书里直接写实现代码块，导致计划与实现混在一起。
- 只列文件名，不写样例和输出。
- 跳过当前基线，直接从理想状态开始写。
