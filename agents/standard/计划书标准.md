# 计划书标准

## 文件说明
- 功能说明：定义计划书的制作顺序、结构、协同、验收和互评口径。
- 使用示例：起草、互评或终验 `ARCHITECTURE/plan/*.md` 前，按本文件核对是否可下发。

## 基本原则
- 计划书是可执行设计文档，不是抽象路线图。
- 计划书先定义完成态，再拆计划内小任务卡；不要从文件清单开始反推目标。
- 计划书必须显式写出公开 `spec / API`、用户确认来源、非目标、禁止修改面、合同真源和验收资产。
- 公开 `API` 变更必须先取得用户明确确认；缺确认时只能列为待确认项。
- 存在争议、冲突、口径不一致或不确定事项时，必须提交用户确认。
- 只做计划书时，正式成稿前必须完成不少于 `3` 个对象的可追溯讨论记录。
- 计划阶段只修改计划书、专题 `spec`、互评摘要或直接配套标准文档；不混入实现、测试或合并改动。

## 制作顺序
1. 收敛用户原始诉求：写清目标、非目标、当前痛点和必须看到的输出。
2. 盘点当前基线：列出现有公开合同、实现入口、测试入口、`expectation` 合同资产和已知失败。
3. 完成不少于 `3` 个对象的讨论记录；讨论对象默认来自 `agents-lists.md`，并保留可追溯记录。
4. 将仍影响方案、范围、依赖、公开 `API` 或验收口径的事项整理成待确认项，提交用户拍板。
5. 用户确认后再写可下发正文；未确认前只允许保留讨论稿或待确认项。
6. 写计划级任务和计划内小任务卡，确认每张卡能被执行人直接照做。
7. 完成自检和互评，所有最小需改项收口后再进入管理员推进。

## 计划级任务
- 一个计划书只对应一个计划级 `execute` 任务。
- 计划内 `S1/S2/...` 是任务卡，不创建独立 `TODO` 状态。
- `execute` 必须一次完成计划内规格、实现、测试与验收闭环。
- 计划级任务之后固定按 `review -> 架构复核 / 终验 -> merge/归档` 推进。
- `review` 不通过回到 `execute`；架构复核 / 终验不通过也回到 `execute`。失败修复仍挂原计划级任务链，不另建独立 `refactor` 阶段。
- 计划级任务目标必须是可执行动作，写清对象、公开行为和验收结果。

## 必备结构
- `文档信息`：目标 `spec`、公开 `API`、测试、验收资产、实现入口。
- `计划级任务`：任务名、任务类型、`worktree`、记录文件。
- `评审摘要`：评审结论、评审人、结论摘要、当前状态。
- `终验 / 复验 / 修复复核记录`：结论人、验证基线、执行目录、合同验收摘要、最小阻断项或通过摘要。
- `计划目标`：当前计划要交付的外部可见结果。
- `当前基线`：当前合同、实现、测试、验收资产、失败点和缺口。
- `方案比较与选型`：不采用方案、采用方案、取舍理由、风险和兼容性影响。
- `公开 API 设计`：API 列表紧跟功能简介；class 场景逐条列公开方法签名。
- `完成态定义`：执行完后必须看到的行为、文件和验证结果。
- `验收设计`：pytest、脚本和 `expectation` 合同验收分开写。
- `计划内小任务`：每张卡写清上下文、目标、非目标、模块范围、禁止修改面、合同真源、最小闭环、执行步骤、验收与记录要求。
- `待确认项`：每项写清选项、差异、影响和推荐项。
- `用户确认与协同约束`：记录用户确认来源和不少于 `3` 个讨论对象。

## 当前基线要求
- 写事实，不写愿景。
- 至少核对当前公开 `spec`、文件级 `API 列表`、实现入口、测试入口、合同验收入口和当前失败摘要。
- 若 `expectation` 路径、入口、依赖或输出不明确，必须写成待确认或架构裁定项。
- 若实现与已明确的 `expectation` 冲突，默认由执行链收口实现；不自动升级为修改 `expectation`。

## 公开 API 要求
- `API 列表` 必须紧跟功能简介，只做快速索引。
- 函数写参数签名、返回值和稳定错误语义。
- class 场景列 `class` 构造签名，并逐条列公开方法签名。
- 工具入口写命令参数、默认值、输出和失败文本。
- include API 写命名空间、函数 / 宏签名和稳定输出格式。
- 未经用户确认的 API 只能列入待确认项，不得写成最终设计。

## 计划内小任务卡
- 每张卡必须自包含，执行人只读该卡即可知道为什么做、做什么、不做什么、怎么验收、记录写到哪里。
- 小任务目标必须是“可执行动作 + 对象 + 验收结果”。
- `执行步骤` 必须按落地顺序写，不能只写“按计划实现”。
- `最小功能闭环` 必须写出最小可交付行为、实现入口、测试入口和失败边界。
- 模块边界优先按子系统范围写，不把正常实现任务压成逐文件白名单。
- 禁止修改面必须显式列出 `expectation/`、计划书、标准文档、immutable 文件等敏感范围。
- 合同真源必须写清优先级，例如 `spec > pytest > expectation > 当前实现`，或按计划实际情况指定。

## 验收口径
- 计划书列出的命令是最低集合；执行人与审查人仍必须按 diff 反推测试。
- `expectation` 是合同验收资产，不计入 diff 反推测试。
- 终验、复验和终验修复复核必须在最新同步现场运行当前计划正文列为必过合同验收资产的相关 `expectation`。
- 若某个 `expectation` 已被正文收为历史 / 本地只读合同来源说明，或 latest main 明确不存在且正文不再列为必过入口，不得把它当作当前阻断项。
- 用户明确要求全量 `expectation` 时才运行全量。
- 通过与否只基于当前计划正文、公开 `API`、当前验收资产和最新同步现场。
- 历史记录、归档说明、对齐文字、阶段编号和说明一致性，不得单独作为不通过理由，除非影响当前功能或验收资产。

## 互评与自检
- 计划是否有用户确认来源和不少于 `3` 个讨论记录。
- 公开 `API`、非目标、禁止修改面和合同真源是否无歧义。
- 每张小任务卡是否能让执行人直接开工。
- 是否检查 `expectation` 修改权限、跨文件非公开 `API` 调用和测试直连非 API helper。
- 验收命令是否可执行，相关 `expectation` 是否只作为合同验收单列。
- 若仍有任何可执行改进项，互评结论不得写 `通过`。
